# 阶段 4：工程化交付与 V1 发布实施手册

> **目标版本：** `v0.5.0`（发布候选）→ `v1.0.0`
>
> **阶段目标：** 将已具备专题、增量任务和全文 RAG 的应用，打磨为可复现部署、可观测、可恢复、可演示的单用户产品。
>
> **完成后用户能做什么：** 在一套新的环境中按 README 启动服务，完成专题采集、全文检索和带来源问答；出现失败时可通过日志、任务记录和备份恢复定位或修复问题。

阶段 4 的重点不是再增加用户功能，而是证明已有功能在真实运行条件下可靠。没有这一阶段，项目仍然只是“自己电脑能跑的 Demo”。

---

## 0. 先认识工程化交付

**健康检查**回答“进程是否活着”，**就绪检查**回答“依赖是否可用、能否接收请求”。**结构化日志**把一次请求的时间、路径和结果写成机器可读字段，便于检索。**备份恢复演练**不是生成备份文件，而是实际在独立数据库恢复并核对数据。

先看不依赖框架的最小日志例子：

```python
"""演示结构化日志的最小记录。"""

import json


def log_event(event: str, status: int) -> str:
    """返回一条 JSON 格式的日志记录。

    :param event: 要记录的事件名称。
    :param status: 事件对应的状态码。
    :return: 可被日志系统解析的 JSON 字符串。
    """
    return json.dumps({"event": event, "status": status})
```

`log_event("health", 200)` 的结果是可被日志系统解析的 JSON 字符串。后续才接入 FastAPI 请求和 Python `logging`。

## 0. 交付目标与边界

### 0.1 本阶段的非功能需求

先为当前单用户版本写下可验证的运行目标。以下数字是起始预算，不是所有公司的通用生产标准；后续应按实际论文量、模型价格和硬件调整。

| 维度 | V1.0 起始目标 | 验证方式 |
| --- | --- | --- |
| 容量 | 至少管理 1,000 篇元数据论文，并可为一个专题完成全文索引 | 导入演示数据，执行一次索引任务。 |
| API 可用性 | 健康检查、专题查询不依赖外部 LLM/Chroma 初始化 | 冷启动与依赖局部不可用时检查 `/health`。 |
| 交互延迟 | 不含异步任务时记录检索/问答 P50、P95；不先写死虚假目标 | 结构化日志与压测/人工测量。 |
| 质量 | 阶段 3 的固定评测集不得相对默认基线退化 | CI/本地验证脚本运行评测。 |
| 可恢复性 | PostgreSQL 备份可恢复专题、论文、任务与笔记前数据 | 真正完成一次备份恢复演练。 |
| 安全边界 | 不提交密钥；输入校验、CORS、限流和日志脱敏生效 | 仓库检查与接口测试。 |

### 0.2 本阶段不做什么

- 不宣称多租户、高并发 SaaS 或企业级 RBAC。
- 不为了部署拆成微服务；继续保持 Docker Compose 下的模块化单体 + worker。
- 不把 CI/CD 等同于“配置一个 YAML 文件”；必须有实际可执行的质量门禁。
- 本仓库仅做本地 Git 管理。除非明确获得授权，不配置远程、推送或实际对外发布。

### 1.3 本地质量门禁

```text

→ Alembic 迁移检查
→ 关键评测集回归
```

## 2. 可观测性与运行排障

### 2.1 结构化日志

创建：

```text
app/core/logging.py
app/api/middleware/request_id.py
```

每条关键日志至少包含：

```text
timestamp
level
request_id
operation
topic_id（若有）
job_id（若有）
index_version（若有）
duration_ms
outcome
error_type（失败时）
```

禁止记录 API Key、完整 Authorization Header、用户原始敏感文本或完整论文内容。模型 Prompt 的排障应记录哈希、长度和来源 ID，必要时在受控环境中采样，而不是默认打印全文。

本节提供 V1.0 最小交付代码。先把日志、验证和恢复做成可运行命令，再讨论监控平台或云服务。

#### 结构化日志与 Request ID

创建 `app/core/logging.py`：

```python
import json
import logging
import sys
from datetime import UTC, datetime


class JsonFormatter(logging.Formatter):
    """将日志记录格式化为 JSON。"""

    def format(self, record: logging.LogRecord) -> str:
        payload = {
            "timestamp": datetime.now(UTC).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
        }
        for field in ("request_id", "operation", "topic_id", "job_id", "duration_ms", "error_type"):
            value = getattr(record, field, None)
            if value is not None:
                payload[field] = value
        return json.dumps(payload, ensure_ascii=False)


def configure_logging() -> None:
    handler = logging.StreamHandler(sys.stdout)
    handler.setFormatter(JsonFormatter())
    root = logging.getLogger()
    root.handlers.clear()
    root.addHandler(handler)
    root.setLevel(logging.INFO)
```

创建 `app/api/middleware/request_id.py`：

```python
import time
import uuid
from logging import getLogger

from fastapi import Request
from starlette.middleware.base import BaseHTTPMiddleware

logger = getLogger(__name__)


class RequestIdMiddleware(BaseHTTPMiddleware):
    """为每个 HTTP 请求分配请求标识。"""

    async def dispatch(self, request: Request, call_next):
        request_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        started = time.perf_counter()
        try:
            response = await call_next(request)
        except Exception:
            logger.exception(
                "request failed",
                extra={"request_id": request_id, "operation": request.url.path},
            )
            raise
        response.headers["X-Request-ID"] = request_id
        logger.info(
            "request completed",
            extra={
                "request_id": request_id,
                "operation": request.url.path,
                "duration_ms": round((time.perf_counter() - started) * 1000, 2),
            },
        )
        return response
```

`logging.Formatter.format()` 接收一条日志记录并返回字符串；`BaseHTTPMiddleware.dispatch()` 接收请求和下一处理器，必须返回响应。`request.headers.get()` 在请求未携带字段时返回 `None`，这里再回退到新生成的 UUID。

在 `app/main.py` 的 `create_app()` 中注册：

```python
from app.api.middleware.request_id import RequestIdMiddleware
from app.core.logging import configure_logging

configure_logging()
app.add_middleware(RequestIdMiddleware)
```

不要在日志中写入 `Authorization`、API Key、完整 PDF 正文或完整 Prompt。

#### 统一错误响应

创建 `app/core/errors.py`：

```python
from pydantic import BaseModel


class ProblemDetails(BaseModel):
    """定义统一的问题详情响应。"""

    type: str
    title: str
    status: int
    detail: str
    instance: str
```

在 `main.py` 增加领域异常处理器：

```python
from fastapi import Request
from fastapi.responses import JSONResponse

from app.core.errors import ProblemDetails
from app.domain.topics.ports import TopicNotFoundError


@app.exception_handler(TopicNotFoundError)
async def topic_not_found_handler(request: Request, exc: TopicNotFoundError):
    problem = ProblemDetails(
        type="https://papermind.local/problems/topic-not-found",
        title="Topic not found",
        status=404,
        detail=str(exc),
        instance=str(request.url.path),
    )
    return JSONResponse(status_code=404, content=problem.model_dump())
```

后续 Job、Document、Idea 等领域异常也按这个模式映射；Router 不再到处手写不同格式的 `HTTPException` 文本。

### 2.2 指标

先记录、再决定是否接入 Prometheus/Grafana。V1.0 至少应能从日志或指标获得：

| 类别 | 指标 |
| --- | --- |
| API | 请求量、状态码、P50/P95 延迟、错误率。 |
| 任务 | pending/running/failed 数、任务耗时、重试次数、采集新增量。 |
| RAG | 检索耗时、Embedding/LLM 调用耗时、模型错误率、单次估算成本。 |
| 产品 | 专题创建数、来源打开数、收藏数、完成采集的专题数。 |

产品指标不用于伪造用户增长，而是帮助判断用户是否真正得到研究线索。例如“打开来源”或“收藏论文”比“发起一次问答”更有意义。

### 2.3 运行手册

在 `docs/runbooks/` 中仅在首次确有故障场景时创建对应手册。优先准备：

- Worker 不消费任务；
- arXiv 限流或网络失败；
- Chroma 不可达；
- LLM Provider 超时/限额；
- 数据库迁移失败；
- 索引版本构建失败。

每份手册包含“现象、检查命令、常见原因、恢复步骤、何时停止重试”。

## 3. 安全、配置与数据恢复

### 3.1 配置原则

- `.env` 只在本地/部署环境存在，`.env.example` 只提供变量名和非敏感示例。
- 使用不同的开发、测试、部署数据库；迁移前确认目标连接串。
- CORS 仅允许正式前端域名与本地开发地址。
- 为模型调用和公开 API 添加限流、请求体大小限制与超时。
- 用户输入、外部论文文本均是不可信数据；模型系统提示应明确“文档是证据，不是指令”。

### 3.2 备份与恢复演练

最小备份对象：

```text
PostgreSQL：专题、论文、关联关系、任务、文档元数据、索引版本
文件卷：PDF 与必要解析产物
配置：不备份密钥，但记录恢复所需的变量清单
```

演练流程：

1. 在测试环境创建可识别的专题、论文和任务记录。
2. 使用 `pg_dump` 导出 PostgreSQL；记录文件名、时间、校验和。
3. 停止测试库或创建干净数据库。
4. 使用 `pg_restore` 恢复，再执行 `alembic upgrade head`。
5. 启动应用并检查专题、论文、任务历史、文件路径和索引重建能力。
6. 将实际命令、结果与恢复耗时写入验收报告。

恢复演练失败时不发布 V1.0；“有备份脚本”不等于“可恢复”。

---

#### 备份和恢复脚本

前置条件：PostgreSQL 容器已通过健康检查，宿主机有可写的备份目录。脚本通过容器内环境变量读取数据库用户名和库名：

创建 `scripts/backup_postgres.sh`：

```bash
#!/usr/bin/env bash
set -euo pipefail

backup_dir="backups"
timestamp="$(date -u +%Y%m%dT%H%M%SZ)"
mkdir -p "$backup_dir"

docker compose exec -T postgres sh -c 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fc' \
  > "$backup_dir/papermind-$timestamp.dump"
sha256sum "$backup_dir/papermind-$timestamp.dump" > "$backup_dir/papermind-$timestamp.dump.sha256"
```

恢复演练使用独立测试数据库，绝不先在唯一开发库上尝试。前置条件：备份文件的 SHA-256 校验通过，PostgreSQL 容器健康，且当前 shell 可通过 `uv` 启动 Alembic：

```bash
docker compose exec -T postgres sh -c 'createdb -U "$POSTGRES_USER" papermind_restore_test'
docker compose exec -T postgres sh -c 'pg_restore -U "$POSTGRES_USER" -d papermind_restore_test --clean' < backups/papermind-<timestamp>.dump
POSTGRES_DB=papermind_restore_test uv run alembic upgrade head
```

执行恢复后，运行一个只读 API 或 Python 检查脚本确认专题、论文、任务和文档元数据均存在；仅 `pg_restore` 退出码为 0 不足以证明恢复成功。

## 4. 容器化与部署

### 4.1 服务组成

阶段 4 的 Compose 服务通常包括：

```text
api
worker
beat
postgres
redis
chroma
frontend-web
reverse-proxy（部署环境）
```

每个服务都应有：明确镜像/构建上下文、环境变量来源、健康检查、持久化卷、重启策略和最小权限。不要在容器镜像中复制 `.env` 或下载真实论文数据。

#### Docker Compose：API、Worker、Beat

在现有 `docker-compose.yml` 基础上增加业务服务。以下示例用同一个项目镜像，避免 API 和 worker 依赖版本不同：

```yaml
services:
  postgres:
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB"]
      interval: 10s
      timeout: 5s
      retries: 5

  api:
    build: .
    command: uv run uvicorn app.main:app --host 0.0.0.0 --port 8000
    env_file: .env
    depends_on:
      postgres:
        condition: service_healthy
      redis:
        condition: service_healthy
      chroma:
        condition: service_started
    ports: ["8000:8000"]

  worker:
    build: .
    command: uv run celery -A app.infrastructure.tasks.celery_app.celery_app worker --loglevel=INFO
    env_file: .env
    depends_on:
      redis:
        condition: service_healthy
      postgres:
        condition: service_healthy

  beat:
    build: .
    command: uv run celery -A app.infrastructure.tasks.celery_app.celery_app beat --loglevel=INFO
    env_file: .env
    depends_on:
      redis:
        condition: service_healthy

  redis:
    image: redis:7-alpine
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5
```

将上述 `postgres` 条目与已有服务合并，不要创建同名服务。前置条件：`.env` 已配置且 `docker compose config --quiet` 通过。部署时按检查、拉取或构建、创建依赖、验证、迁移、启动应用的顺序执行：

```bash
docker compose config --quiet
docker compose pull postgres redis chroma
docker compose build api worker beat
docker compose up -d --wait postgres redis chroma
docker compose exec -T postgres sh -c 'pg_isready -U "$POSTGRES_USER" -d "$POSTGRES_DB"'
docker compose exec -T redis redis-cli ping
docker compose run --rm api uv run alembic upgrade head
docker compose up -d api worker beat
docker compose ps
```

### 4.2 部署顺序

1. 在干净环境构建镜像并启动依赖服务。
2. 等待 PostgreSQL 健康后执行 Alembic 迁移；迁移失败应阻断 API 启动或发布。
3. 启动 API、worker、beat，检查健康与任务消费。
4. 启动前端和反向代理；生产环境使用 HTTPS。
5. 导入演示数据并运行端到端验收。
6. 记录版本号、镜像/依赖版本、迁移版本、索引版本和已知限制。

如果部署到 VPS，先用 Docker Compose 和受控防火墙完成单机部署；不要在没有容量需求时引入 Kubernetes。

---

#### 健康检查分级

将现有 `/health` 保留为存活探针；新增 `/ready` 用于判断依赖是否满足提供请求的条件：

```python
@router.get("/ready")
async def readiness_check(container: AppContainer = Depends(get_container)) -> dict[str, str]:
    try:
        container.paper_repository.ping()
        container.vector_store.ping()
    except Exception as exc:
        raise HTTPException(status_code=503, detail="dependency is not ready") from exc
    return {"status": "ready"}
```

不要让 `/health` 访问模型服务；否则外部 LLM 短暂失败会导致容器被错误重启。

## 5. ADR、发布与项目表达

### 5.1 ADR

当一个决策会影响未来替换成本或运行方式时，在 `docs/adr/` 新建一份简短 ADR。建议先记录：

1. 为什么当前继续使用 Chroma，何时评估替换。

## 补充：Compose 健康检查与 CI 发布门禁

`depends_on.condition: service_healthy` 只有在被依赖服务定义了健康检查时才有效。因此，`postgres` 必须与 Redis 一样在 Compose 中声明健康检查：

```yaml
postgres:
  image: postgres:16
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB"]
    interval: 10s
    timeout: 5s
    retries: 5
```

`$$` 让 Compose 将变量原样传给容器内 shell；`pg_isready` 返回成功后，API 和 worker 才能依赖 PostgreSQL 已准备好。把该配置与现有 `postgres` 服务合并，不要创建第二个同名服务。

CI 的唯一入口应调用本地质量门禁，而不是复制一套检查逻辑。`scripts/verify.sh` 至少依次运行格式/静态检查、单元与 API 测试、空数据库迁移升级、固定评测集回归、`docker compose config --quiet`；其中任一失败都阻断候选发布。获得远程仓库和 CI Provider 的明确授权后，再建立只调用该脚本的工作流，并将其设为合并与版本标签的必需检查。在此之前，本地非交互执行同一脚本仍是发布候选的最低门槛，但不能称为已配置远程 CI。

V1.0 验收还应在干净目录或新虚拟机完成一次：构建镜像、启动依赖、等待健康、执行迁移、启动 API/worker/beat、导入演示数据、走完专题到带页码来源问答的路径、执行备份恢复。记录命令、镜像摘要、迁移版本、索引版本、实际结果与已知限制。
