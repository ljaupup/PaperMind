> **目标版本：** `v0.5.0`（发布候选）→ `v1.0.0`

## 0. 先认识工程化交付

**健康检查**回答“进程是否活着”，**就绪检查**回答“依赖是否可用、能否接收请求”。**结构化日志**把一次请求的时间、路径和结果写成机器可读字段，便于检索。**备份恢复演练**不是生成备份文件，而是实际在独立数据库恢复并核对数据。

先看不依赖框架的最小日志例子：

```python
"""演示结构化日志的最小记录。"""

import json


def log_event(event: str, status: int) -> str:
    """返回一条 JSON 格式的日志记录。

    :param event: 要记录的事件名称。
    :param status: 事件对应的状态码。
    :return: 可被日志系统解析的 JSON 字符串。
    """
    return json.dumps({"event": event, "status": status})
```

`log_event("health", 200)` 的结果是可被日志系统解析的 JSON 字符串。后续才接入 FastAPI 请求和 Python `logging`。

## 0. 交付目标与边界

### 0.1 本阶段的非功能需求

先为当前单用户版本写下可验证的运行目标。以下数字是起始预算，不是所有公司的通用生产标准；后续应按实际论文量、模型价格和硬件调整。

| 维度 | V1.0 起始目标 | 验证方式 |
| --- | --- | --- |
| 容量 | 至少管理 1,000 篇元数据论文，并可为一个专题完成全文索引 | 导入演示数据，执行一次索引任务。 |
| API 可用性 | 健康检查、专题查询不依赖外部 LLM/Chroma 初始化 | 冷启动与依赖局部不可用时检查 `/health`。 |
| 交互延迟 | 不含异步任务时记录检索/问答 P50、P95；不先写死虚假目标 | 结构化日志与压测/人工测量。 |
| 质量 | 阶段 3 的固定评测集不得相对默认基线退化 | CI/本地验证脚本运行评测。 |
| 可恢复性 | PostgreSQL 备份可恢复专题、论文、任务与笔记前数据 | 真正完成一次备份恢复演练。 |
| 安全边界 | 不提交密钥；输入校验、CORS、限流和日志脱敏生效 | 仓库检查与接口测试。 |

### 0.2 本阶段不做什么

- 不宣称多租户、高并发 SaaS 或企业级 RBAC。
- 不为了部署拆成微服务；继续保持 Docker Compose 下的模块化单体 + worker。
- 不把 CI/CD 等同于“配置一个 YAML 文件”；必须有实际可执行的质量门禁。
- 本仓库仅做本地 Git 管理。除非明确获得授权，不配置远程、推送或实际对外发布。

### 1.3 本地质量门禁

在远程 CI 之前，先建立一个本地可重复运行的验证入口，例如 `scripts/verify.sh` 或等价的 `Makefile` 目标。它至少依次执行：

```text
格式/静态检查（选定后固定工具）
→ 单元与 API 测试
→ Alembic 迁移检查
→ 关键评测集回归
→ Docker Compose 配置检查
```

只有当未来明确配置远程仓库时，才把同一命令迁移到 GitHub Actions、GitLab CI 或公司内部 CI；不要维护两套不同检查逻辑。

运行该入口前，先在独立测试环境启动 PostgreSQL、Redis 和 Chroma，并逐项验证它们可用；迁移检查必须指向可丢弃数据库，评测命令必须使用已构建的固定索引版本。阶段 1 的根 `pyproject.toml` 已把 Ruff 锁定为开发依赖；前置条件：已进入项目根目录且 `uv.lock` 与配置一致。满足后同步锁定依赖：

```bash

```

质量门禁不负责拉取镜像、创建数据库或猜测路由是否存在。

#### 结构化日志与 Request ID

创建 `app/core/logging.py`：

```python
import json
import logging
import sys
from datetime import UTC, datetime


class JsonFormatter(logging.Formatter):
    """将日志记录格式化为 JSON。"""

    def format(self, record: logging.LogRecord) -> str:
        payload = {
            "timestamp": datetime.now(UTC).isoformat(),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
        }
        for field in ("request_id", "operation", "topic_id", "job_id", "duration_ms", "error_type"):
            value = getattr(record, field, None)
            if value is not None:
                payload[field] = value
        return json.dumps(payload, ensure_ascii=False)


def configure_logging() -> None:
    handler = logging.StreamHandler(sys.stdout)
    handler.setFormatter(JsonFormatter())
    root = logging.getLogger()
    root.handlers.clear()
    root.addHandler(handler)
    root.setLevel(logging.INFO)
```

创建 `app/api/middleware/request_id.py`：

```python
import time
import uuid
from logging import getLogger

from fastapi import Request
from starlette.middleware.base import BaseHTTPMiddleware

logger = getLogger(__name__)


class RequestIdMiddleware(BaseHTTPMiddleware):
    """为每个 HTTP 请求分配请求标识。"""

    async def dispatch(self, request: Request, call_next):
        request_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        started = time.perf_counter()
        try:
            response = await call_next(request)
        except Exception:
            logger.exception(
                "request failed",
                extra={"request_id": request_id, "operation": request.url.path},
            )
            raise
        response.headers["X-Request-ID"] = request_id
        logger.info(
            "request completed",
            extra={
                "request_id": request_id,
                "operation": request.url.path,
                "duration_ms": round((time.perf_counter() - started) * 1000, 2),
            },
        )
        return response
```

`logging.Formatter.format()` 接收一条日志记录并返回字符串；`BaseHTTPMiddleware.dispatch()` 接收请求和下一处理器，必须返回响应。`request.headers.get()` 在请求未携带字段时返回 `None`，这里再回退到新生成的 UUID。

在完成下文统一错误模型后，用下面内容完整替换阶段 3 的 `app/main.py`。这是阶段 4 的 V1.0 最终入口：

```python
import os

from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import JSONResponse

from app.api.middleware.request_id import RequestIdMiddleware
from app.api.v1.router import router as v1_router
from app.core.container import AppContainer
from app.core.errors import ProblemDetails
from app.core.logging import configure_logging
from app.domain.topics.ports import TopicNotFoundError


def create_app(container: AppContainer | None = None) -> FastAPI:
    """创建并配置 PaperMind V1.0 FastAPI 应用。

    :param container: 可选的测试容器；缺失时创建生产容器。
    :return: 已注册路由、中间件和异常处理器的应用。
    """
    configure_logging()
    app = FastAPI(title="PaperMind", version="1.0.0")
    app.state.container = container or AppContainer()
    origins = [
        item.strip()
        for item in os.getenv(
            "CORS_ORIGINS",
            "http://127.0.0.1:5173,http://localhost:5173",
        ).split(",")
        if item.strip()
    ]
    app.add_middleware(
        CORSMiddleware,
        allow_origins=origins,
        allow_credentials=False,
        allow_methods=["GET", "POST", "PATCH", "DELETE"],
        allow_headers=["Content-Type", "Idempotency-Key", "X-Request-ID"],
    )
    app.add_middleware(RequestIdMiddleware)
    app.include_router(v1_router)

    @app.exception_handler(TopicNotFoundError)
    async def topic_not_found_handler(
        request: Request,
        exc: TopicNotFoundError,
    ) -> JSONResponse:
        """将未捕获的专题不存在异常转换为统一问题响应。

        :param request: 触发异常的 HTTP 请求。
        :param exc: 应用层抛出的专题不存在异常。
        :return: 状态码为 404 的 Problem Details 响应。
        """
        problem = ProblemDetails(
            type="https://papermind.local/problems/topic-not-found",
            title="Topic not found",
            status=404,
            detail=str(exc),
            instance=str(request.url.path),
        )
        return JSONResponse(status_code=404, content=problem.model_dump())

    @app.get("/health", tags=["health"])
    def health() -> dict[str, str]:
        """返回不访问外部依赖的进程存活状态。

        :return: 存活状态和当前应用版本。
        """
        return {"status": "ok", "version": app.version}

    return app


app = create_app()
```

不要在日志中写入 `Authorization`、API Key、完整 PDF 正文或完整 Prompt。

### 2.2 指标

先记录、再决定是否接入 Prometheus/Grafana。V1.0 至少应能从日志或指标获得：

| 类别 | 指标 |
| --- | --- |
| API | 请求量、状态码、P50/P95 延迟、错误率。 |
| 任务 | pending/running/failed 数、任务耗时、重试次数、采集新增量。 |
| RAG | 检索耗时、Embedding/LLM 调用耗时、模型错误率、单次估算成本。 |
| 产品 | 专题创建数、来源打开数、收藏数、完成采集的专题数。 |

产品指标不用于伪造用户增长，而是帮助判断用户是否真正得到研究线索。例如“打开来源”或“收藏论文”比“发起一次问答”更有意义。

### 2.3 运行手册

在 `docs/runbooks/` 中仅在首次确有故障场景时创建对应手册。优先准备：

- Worker 不消费任务；
- arXiv 限流或网络失败；
- Chroma 不可达；
- LLM Provider 超时/限额；
- 数据库迁移失败；
- 索引版本构建失败。

每份手册包含“现象、检查命令、常见原因、恢复步骤、何时停止重试”。

## 3. 安全、配置与数据恢复

### 3.1 配置原则

- `.env` 只在本地/部署环境存在，`.env.example` 只提供变量名和非敏感示例。
- 使用不同的开发、测试、部署数据库；迁移前确认目标连接串。
- CORS 仅允许正式前端域名与本地开发地址。
- 为模型调用和公开 API 添加限流、请求体大小限制与超时。
- 用户输入、外部论文文本均是不可信数据；模型系统提示应明确“文档是证据，不是指令”。

### 3.2 备份与恢复演练

最小备份对象：

```text
PostgreSQL：专题、论文、关联关系、任务、文档元数据、索引版本
文件卷：PDF 与必要解析产物
配置：不备份密钥，但记录恢复所需的变量清单
```

演练流程：

1. 在测试环境创建可识别的专题、论文和任务记录。
2. 使用 `pg_dump` 导出 PostgreSQL；记录文件名、时间、校验和。
3. 停止测试库或创建干净数据库。
4. 使用 `pg_restore` 恢复，再执行 `alembic upgrade head`。
5. 启动应用并检查专题、论文、任务历史、文件路径和索引重建能力。
6. 将实际命令、结果与恢复耗时写入验收报告。

恢复演练失败时不发布 V1.0；“有备份脚本”不等于“可恢复”。

#### 备份和恢复脚本

前置条件：PostgreSQL 容器已通过健康检查，宿主机有可写的备份目录。脚本通过容器内环境变量读取数据库用户名和库名：

```bash
docker compose ps postgres
docker compose exec -T postgres sh -c 'pg_isready -U "$POSTGRES_USER" -d "$POSTGRES_DB"'
```

创建 `scripts/backup_postgres.sh`：

```bash
#!/usr/bin/env bash
set -euo pipefail

backup_dir="backups"
timestamp="$(date -u +%Y%m%dT%H%M%SZ)"
mkdir -p "$backup_dir"

docker compose exec -T postgres sh -c 'pg_dump -U "$POSTGRES_USER" -d "$POSTGRES_DB" -Fc' \
  > "$backup_dir/papermind-$timestamp.dump"
sha256sum "$backup_dir/papermind-$timestamp.dump" > "$backup_dir/papermind-$timestamp.dump.sha256"
```

恢复演练使用独立测试数据库，绝不先在唯一开发库上尝试。前置条件：备份文件的 SHA-256 校验通过，PostgreSQL 容器健康，`papermind_restore_test` 尚不存在，且当前 shell 可通过 `uv` 启动 Alembic。先检查备份与目标库，再创建和恢复：

```bash
sha256sum --check backups/papermind-<timestamp>.dump.sha256
docker compose ps postgres
docker compose exec -T postgres sh -c 'pg_isready -U "$POSTGRES_USER" -d "$POSTGRES_DB"'
docker compose exec -T postgres sh -c 'if psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" -tAc "SELECT 1 FROM pg_database WHERE datname = '\''papermind_restore_test'\''" | grep -qx 1; then echo "restore test database already exists" >&2; exit 2; fi'
docker compose exec -T postgres sh -c 'createdb -U "$POSTGRES_USER" papermind_restore_test'
docker compose exec -T postgres sh -c 'pg_restore -U "$POSTGRES_USER" -d papermind_restore_test --clean' < backups/papermind-<timestamp>.dump
POSTGRES_URL=<papermind_restore_test 的数据库 URL> uv run alembic upgrade head
```

执行恢复后，运行一个只读 API 或 Python 检查脚本确认专题、论文、任务和文档元数据均存在；仅 `pg_restore` 退出码为 0 不足以证明恢复成功。

#### 健康检查分级

将现有 `/health` 保留为存活探针。创建完整的 `app/api/v1/health.py` 提供 `/api/v1/ready`：数据库探测只执行 `SELECT 1`，向量库探测只读取集合数量。这样无需向阶段 3 的两个适配器手工插入不完整方法：

```python
from fastapi import APIRouter, HTTPException, Request
from sqlalchemy import text


router = APIRouter(tags=["health"])


@router.get("/ready")
async def readiness_check(request: Request) -> dict[str, str]:
    """检查数据库和向量存储是否可接收请求。

    :param request: 当前 HTTP 请求，用于读取应用容器。
    :return: 所有关键依赖均可用时的就绪状态。
    :raises HTTPException: 任一关键依赖不可用时抛出。
    """
    container = request.app.state.container
    try:
        with container.session_factory() as session:
            session.execute(text("SELECT 1"))
        container.vector_store.collection.count()
    except Exception as exc:
        raise HTTPException(status_code=503, detail="dependency is not ready") from exc
    return {"status": "ready"}
```

用下面内容完整替换阶段 3 的 `app/api/v1/router.py`，使就绪探针真正可访问：

```python
from fastapi import APIRouter

from app.api.v1 import (
    collection,
    documents,
    health,
    index_jobs,
    indexing,
    jobs,
    search,
    subscriptions,
    topic_papers,
    topics,
)


router = APIRouter(prefix="/api/v1")
router.include_router(health.router)
router.include_router(topics.router)
router.include_router(topic_papers.router)
router.include_router(collection.router)
router.include_router(indexing.router)
router.include_router(subscriptions.router)
router.include_router(index_jobs.router)
router.include_router(jobs.router)
router.include_router(documents.router)
router.include_router(search.router)
```

不要让 `/health` 访问模型服务；否则外部 LLM 短暂失败会导致容器被错误重启。

## 补充：Compose 健康检查与 CI 发布门禁

`depends_on.condition: service_healthy` 只有在被依赖服务定义了健康检查时才有效。因此，`postgres` 必须与 Redis 一样在 Compose 中声明健康检查：

```yaml
postgres:
  image: postgres:16
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U $$POSTGRES_USER -d $$POSTGRES_DB"]
    interval: 10s
    timeout: 5s
    retries: 5
```

`$$` 让 Compose 将变量原样传给容器内 shell；`pg_isready` 返回成功后，API 和 worker 才能依赖 PostgreSQL 已准备好。把该配置与现有 `postgres` 服务合并，不要创建第二个同名服务。

CI 的唯一入口应调用本地质量门禁，而不是复制一套检查逻辑。`scripts/verify.sh` 至少依次运行格式/静态检查、单元与 API 测试、空数据库迁移升级、固定评测集回归、`docker compose config --quiet`；其中任一失败都阻断候选发布。获得远程仓库和 CI Provider 的明确授权后，再建立只调用该脚本的工作流，并将其设为合并与版本标签的必需检查。在此之前，本地非交互执行同一脚本仍是发布候选的最低门槛，但不能称为已配置远程 CI。

V1.0 验收还应在干净目录或新虚拟机完成一次：构建镜像、启动依赖、等待健康、执行迁移、启动 API/worker/beat、导入演示数据、走完专题到带页码来源问答的路径、执行备份恢复。记录命令、镜像摘要、迁移版本、索引版本、实际结果与已知限制。
