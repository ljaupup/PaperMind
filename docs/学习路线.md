# PaperMind 完整项目学习路线

> **项目定位：** 面向研究者的预印本研究情报与证据对话应用。
>
> **首要目标：** 将 PaperMind 打磨为一个完整、可演示、可部署的 AI 应用项目，用于求职作品集；论文阅读和 Idea 发现是它有真实用户价值的业务场景，而不是把技术点拼在一起的练习题。
>
> **使用方式：** 本文不是从零入门课程，也不设固定周数。它以当前已完成的 MVP 为起点，按产品价值和工程依赖排列后续学习与落地内容：每完成一个阶段，就应得到可运行的功能、可验证的质量证据和可在简历中说明的工程能力。

相关历史记录见 [7 月 AI 应用开发实战文档](./PaperMind-7月底AI应用开发实战文档.md)。该文档记录了从零搭建当前 MVP 的过程，作为已完成阶段的学习与实施档案保留，不在本路线中重复改写。

## 一、先明确要做成什么

### 1. 产品一句话

用户围绕一个研究主题，持续获取 arXiv 预印本；在自己的论文工作区中完成筛选、阅读、检索、比较与对话，并把回答依据、自己的判断和待验证的研究假设沉淀为可回溯的记录。

PaperMind 不负责判定一篇预印本“正确”或“错误”。它应始终提示预印本未经同行评审，并把论文、页码或片段作为用户判断和后续验证的证据入口。

### 2. 核心用户闭环

```text
创建研究专题
  → 自动/手动采集新论文
  → 筛选、收藏、标记阅读状态
  → 查看论文与全文证据
  → 检索、比较和追问
  → 记录笔记、问题与 Idea 卡片
  → 在下一轮新论文到来时继续验证或修正判断
```

这个闭环比“输入问题、返回一段 RAG 答案”更接近实际应用：它有明确用户、持久数据、任务状态和可持续使用的理由。

### 3. 第一个可演示版本（V1.0）边界

V1.0 的目标是让一位研究者能独立、完整地完成一次可信的专题研究流程，而不是在第一个版本塞入所有设想。它采用**单用户工作区**，暂不实现登录、多人协作和复杂权限；这些能力在核心闭环稳定后再进入后续版本。

| V1.0 必须完成 | 明确延后 |
| --- | --- |
| 创建专题，配置关键词与采集范围。 | 登录、多人协作、组织和细粒度权限。 |
| 手动或定时增量采集，查看任务进度与失败原因。 | 跨来源的通用学术搜索与推荐系统。 |
| 浏览、筛选、收藏和查看论文全文。 | Idea 卡片的完整验证流与自动推荐。 |
| 基于全文进行检索、问答，并跳转到可定位的来源。 | 知识图谱、多智能体、微服务化等非核心扩展。 |
| 在可复现环境中部署、演示和验收。 | 为扩展而扩展的模型或平台能力。 |

论文比较、笔记和证据化 Idea 卡片属于 V1.1 的重点：它们应建立在 V1.0 的全文证据与检索评测已经可靠的前提上。

### 4. 求职作品集的完成标准

在简历或面试中，PaperMind 应能同时证明以下能力，而不是只展示一次 API 调用：

| 维度 | 应能展示的内容 |
| --- | --- |
| 业务建模 | 研究专题、论文、采集任务、工作区、笔记、Idea 卡片等实体与状态流转。 |
| AI 能力 | 可解释的 RAG 链路、证据来源、检索评测、拒答边界和模型配置。 |
| 后端工程 | API 设计、PostgreSQL 数据建模、异步任务、权限、异常处理与可观测性。 |
| 前端产品 | 正式 Web 界面、可用的阅读/检索/对话体验，以及合理的空状态和错误反馈。 |
| 交付质量 | 自动化测试、CI、容器化、部署文档、演示数据和可复现的验收过程。 |

### 5. 范围边界

- 不把 PaperMind 做成通用低代码平台、通用聊天机器人或另一个 Dify/RAGFlow。
- 不以“生成结论”替代研究者核对原文；回答必须优先提供可定位的来源。
- 不在没有评测集和证据的情况下，把“换模型”当成检索优化。
- 不为简历虚构指标或未完成能力；每一项写入简历的内容都要有代码、文档或演示可支撑。

## 二、当前基线：已经具备什么

当前仓库已经完成一个可工作的后端 RAG MVP。它是后续演进的地基，而不是需要推倒重来：

| 模块 | 当前能力 |
| --- | --- |
| 采集 | 按关键词从 arXiv 获取论文元数据和摘要。 |
| 存储 | 可选择本地 JSON 或 PostgreSQL 保存论文元数据。 |
| 索引与检索 | 对标题、摘要分块，使用 ChromaDB 与 Embedding 建立向量检索。 |
| 问答 | FastAPI 提供检索与带来源片段的 RAG 问答；LLM 可接 mock、DeepSeek 或 SiliconFlow。 |
| 模型与基础设施 | 支持 SiliconFlow Embedding、Docker Compose 下的 PostgreSQL 和 ChromaDB。 |
| 验证入口 | 已有 pytest 与 Gradio 测试界面，可人工核对检索片段、问答与来源。 |

当前最关键的限制也要如实保留：论文内容主要是标题与摘要，检索尚未经过系统评测，Gradio 是调试界面而不是正式产品界面，尚无账户、专题、任务、笔记和部署运维闭环。

## 三、路线总览与优先级

按以下顺序推进。前两个阶段优先建立可用产品骨架；检索质量和工程交付随后形成可信度；研究工作流功能最后体现业务差异化。

```text
已完成：后端 RAG MVP
    ↓
阶段 1：最小持久化与正式前端
    ↓
阶段 2：专题订阅与增量任务
    ↓
阶段 3：全文解析与可评测检索
    ↓
阶段 4：工程化交付并发布 V1.0
    ↓
阶段 5：V1.1 研究工作流与证据化 Idea
```

| 优先级 | 阶段 | 解决的问题 | 完成后可展示的成果 |
| --- | --- | --- | --- |
| P0 | 1. 最小持久化与正式前端 | MVP 只有接口和调试页，且缺少支撑页面的稳定领域数据。 | 单用户工作区、研究专题、论文列表、检索和对话页面。 |
| P0 | 2. 专题订阅与增量任务 | 论文库没有持续更新机制，长耗时操作不能可靠追踪。 | 订阅规则、采集任务、去重、状态与任务历史。 |
| P0 | 3. 全文与检索评测 | 仅用摘要难以支撑高质量回答。 | PDF 全文、可定位来源、评测集与量化改进。 |
| P0 | 4. 工程化交付与 V1.0 发布 | 本地能跑不等于可复现、可演示、可维护。 | CI、监控、备份恢复、部署、验收和发布材料。 |
| P1 | 5. V1.1 研究工作流 | 对话结果不能沉淀为下一步研究行动。 | 比较、笔记、阅读状态、Idea 卡片和证据链。 |

## 四、分阶段学习与实施路线

每个阶段都按“理解必要概念 → 做最小验证 → 集成到 PaperMind → 验收并记录”的顺序执行。学习范围服务于当前阶段，不为完整课程体系而学习。

### 阶段 1：最小持久化与正式前端

**目标：** 先建立 V1.0 所需的最小领域数据和迁移能力，再从“后端 RAG Demo”变为可连续操作的单用户应用。Gradio 保留为调试与验收工具，正式使用界面单独建设。

**实施手册：** [阶段 1：最小持久化与正式前端](./阶段1-最小持久化与正式前端实施手册.md)

#### 需要学习并落地的内容

| 主题 | 学习重点 | PaperMind 落地 |
| --- | --- | --- |
| 前端工程 | TypeScript、组件状态、路由、表单、请求与错误处理。推荐 React + Vite 或 Next.js，二选一后保持稳定。 | 研究专题列表、论文列表/详情、检索页、对话页。 |
| 最小领域模型 | 单用户工作区、研究专题、论文与专题论文关系、收藏等最小实体；关系约束与审计字段。 | 页面和 API 不再依赖临时状态，专题、论文筛选和收藏可稳定保存。 |
| 数据迁移 | Alembic 或等价工具的迁移版本管理、初始化与回滚。 | 开发、测试和部署环境可按同一迁移链创建或升级数据库。 |
| RESTful API 契约 | 资源建模、HTTP 方法与状态码、API 版本、分页/筛选、幂等键、统一错误响应。 | 从 `v0.2.0` 起将业务接口置于 `/api/v1`：专题、论文、会话和任务是资源；创建资源返回 `201`，异步采集/索引任务返回 `202` 与可查询的任务资源。 |
| 用户体验 | 加载、空状态、失败重试、来源展开、长任务进度。 | 让用户能理解“系统正在做什么、回答依据是什么、下一步能做什么”。 |
| 身份边界 | 单用户资源归属与未来认证边界。 | V1.0 使用单用户工作区；登录、JWT 或第三方认证在核心闭环稳定后再接入。 |

#### 推荐的 `app/` 分层

从 `v0.2.0` 开始采用模块化单体的分层结构。当前只创建已有能力实际使用的文件；专题、任务、会话等目录在相应业务实现时再新增，不为了目录完整性提前建立空模块。

```text
app/
├── main.py
├── core/
├── api/
├── application/
├── domain/
├── schemas/
└── infrastructure/
```

| 目录/文件 | 职责 | 依赖边界 |
| --- | --- | --- |
| `main.py` | 创建 FastAPI 应用、注册路由和应用生命周期。 | 不编排采集、索引或问答业务逻辑。 |
| `core/` | 配置、依赖装配、日志和统一错误模型等横切能力。 | 不承载具体业务或 Provider 实现。 |
| `api/` | `/api/v1` 路由、HTTP 请求校验、状态码与响应。 | 只调用 `application/`，不直接访问数据库、Chroma 或模型服务。 |
| `application/` | 采集、索引、检索、问答等用例编排与事务边界。 | 依赖 `domain/` 的模型和端口，不绑定具体基础设施。 |
| `domain/` | 论文、文本块等业务模型，以及仓储、向量检索、Embedding、LLM 的抽象端口。 | 不导入 FastAPI、SQLAlchemy、Chroma 或具体模型 SDK。 |
| `schemas/` | API 的 Pydantic 请求/响应 DTO。 | 不替代领域模型，也不放 ORM 模型。 |
| `infrastructure/` | arXiv、PostgreSQL/JSON、Chroma、Embedding、LLM 等具体适配器。 | 实现 `domain/` 端口，由 `core/` 装配。 |

依赖方向固定为：`api → application → domain`，`infrastructure → domain`；`main.py`/`core` 负责将基础设施实现注入应用服务。暂不创建泛用 `utils/`，避免把业务逻辑沉入无边界的工具目录；确有跨业务的纯函数时再按用途加入 `shared/`。

#### 新功能的开发顺序

新功能不从 FastAPI Router 开始写。每次以一个可验收的业务资源或用例为单位，按以下顺序推进；没有实际业务用途的目录和占位文件不提前创建。

| 顺序 | 位置 | 先完成什么 | 验证重点 |
| --- | --- | --- | --- |
| 0 | 学习路线、README 或 ADR | 用户问题、资源边界、RESTful 路径、状态码、非目标与验收条件。 | 能说明为何需要该功能，以及何时算完成。 |
| 1 | `domain/<feature>/models.py` | 业务模型、状态和不变量，例如 `Topic`、`CollectionJob`。 | 不依赖 FastAPI、ORM 或具体 Provider。 |
| 2 | `domain/<feature>/ports.py` | 仓储、任务、检索等抽象端口。 | 应用层可用 Fake 实现测试，不依赖真实数据库或模型。 |
| 3 | `tests/` 与 `application/<feature>_service.py` | 先写应用服务的关键用例测试，再实现创建、查询、状态转换等编排。 | 业务规则、幂等性、失败路径和状态转换。 |
| 4 | `infrastructure/` | ORM 模型、Alembic 迁移、Repository、外部 API 或后台任务适配器。 | 真实实现满足端口；迁移可升级、任务可追踪。 |
| 5 | `core/container.py` | 将具体基础设施实现注入应用服务。 | 服务无需修改即可在 Fake 和真实实现间切换。 |
| 6 | `schemas/<feature>.py` | 请求/响应 DTO、分页和统一错误模型。 | DTO 仅表达 HTTP 契约，不替代领域模型。 |
| 7 | `api/routers/<feature>.py` | Router 调用应用服务，返回正确 HTTP 状态码。 | Router 不直接访问数据库、向量库或模型服务。 |
| 8 | `api/` 聚合路由与 `main.py` | 注册接口、更新 OpenAPI 和依赖配置。 | 接口可从 `/docs` 发现，应用启动不引入无关外部连接。 |
| 9 | `tests/`、README、验收记录 | API 集成测试、人工路径验收、更新接口说明和版本记录。 | 请求校验、状态码、响应、失败降级和回归测试均通过。 |

以未来“研究专题”为例，文件创建顺序是：

```text
domain/topics/models.py
  → domain/topics/ports.py
  → tests/test_topic_service.py + application/topic_service.py
  → infrastructure/persistence/ 的 Topic ORM、迁移和 Repository
  → core/container.py
  → schemas/topics.py
  → api/routers/topics.py
  → API 路由聚合与 tests/test_api_topics.py
```

“采集任务”也遵循同一顺序，但应将 `POST /api/v1/topics/{topic_id}/collection-jobs` 建模为创建任务资源，返回 `202 Accepted`；任务完成情况通过 `GET /api/v1/collection-jobs/{job_id}` 查询。不要把长耗时采集、下载、解析或索引直接放在 Web 请求进程中。

#### 最小交付与验收

- 能在 Web 页面创建一个本地研究专题，并在专题中查看论文列表。
- 研究专题、论文归属与收藏等最小数据有迁移、唯一约束和自动化测试，不以浏览器内临时状态代替持久化。
- 用户可在页面执行检索和问答；回答下方展示论文标题、链接与来源片段。
- 接口对非法参数、空结果、模型失败和索引未构建给出清晰反馈。
- 专题、论文、会话和任务使用资源导向的 RESTful 路径；列表支持分页与筛选，创建/更新/删除遵循 `POST`、`PATCH`、`DELETE` 的语义，错误响应采用统一格式。
- 保留 Gradio 作为开发者验收页，不把它包装成正式生产前端。

#### 这一阶段应能说明的能力

“独立完成 FastAPI + TypeScript 前端的 AI 应用前后端联调，将 RAG API 设计为支持分页、状态反馈与来源可追溯的研究工作区界面。”

### 阶段 2：专题订阅与增量任务

**目标：** 让论文采集从一次性脚本变成可持续运行、可追溯的业务流程。

**实施手册：** [阶段 2：专题订阅与增量任务](./阶段2-专题订阅与增量任务实施手册.md)

#### 需要学习并落地的内容

| 主题 | 学习重点 | PaperMind 落地 |
| --- | --- | --- |
| 订阅与任务建模 | 订阅规则、任务状态机、调度策略、失败原因和审计字段。 | 在既有专题基础上增加订阅规则、采集任务与索引任务，明确状态转换和未来用户归属边界。 |
| 任务架构决策 | Web 请求与长任务的边界、调度器/队列选择、worker 部署、重试与死信处理。 | 在下载、解析和索引前记录任务框架的取舍，避免将长耗时工作留在 API 请求进程中。 |
| 异步任务 | 队列/调度、幂等、重试、限流、任务状态。 | 关键词订阅、定期拉取 arXiv、PDF 下载和索引构建进入后台任务。 |
| 数据质量 | DOI/arXiv ID 去重、更新时间、失败记录、来源保留。 | 同一论文不会被重复写入；用户能看到一条任务是否成功、失败原因和新增数量。 |

#### 最小交付与验收

- 创建专题时可配置关键词、分类、最大返回量和更新频率。
- 系统可以手动触发一次增量采集；重复执行不会产生重复论文。
- 采集任务具有 `pending/running/succeeded/failed` 等明确状态，并记录开始时间、结束时间、统计结果与错误信息。
- 至少为去重规则、任务状态转换、幂等重试和调度入口编写自动化测试。

#### 这一阶段应能说明的能力

“基于 PostgreSQL 建模研究专题和论文资产，设计幂等的 arXiv 增量采集任务与状态追踪机制，支持持续更新而非一次性导入。”

### 阶段 3：全文解析、检索优化与可量化评测

**目标：** 先建立可重复的评测基线，再以证据驱动优化；这是提升 RAG 效果的核心阶段。

**实施手册：** [阶段 3：全文解析与 RAG 评测](./阶段3-全文解析与RAG评测实施手册.md)

#### 需要学习并落地的内容

| 主题 | 学习重点 | PaperMind 落地 |
| --- | --- | --- |
| 全文处理 | PDF 下载、解析失败处理、章节结构、页码、清洗与版本记录。 | 从仅索引标题/摘要升级为正文分块；来源至少包含论文、章节或页码。 |
| 数据来源与使用边界 | 下载来源、许可/使用说明、访问频率、保留策略与不可用资源处理。 | 保留 arXiv 来源和全文获取状态；尊重来源约束，全文不可用时不阻断论文元数据和摘要的使用。 |
| 数据与索引版本 | PDF 来源、下载时间、解析器版本、分块配置、Embedding 模型与索引版本。 | 检索结果和评测结果可追溯到具体数据、解析和索引配置，支持重建与回归比较。 |
| 分块策略 | 结构化分块、重叠窗口、标题继承、块长度与元数据。 | 比较摘要块、段落块、章节块等策略，记录配置与结果。 |
| 混合检索 | 向量检索、关键词检索、metadata filter、候选合并。 | 可按专题、日期、分类、已收藏状态过滤，并融合语义与关键词召回。 |
| 重排序 | cross-encoder/LLM rerank 的适用边界、延迟与成本。 | 对 Top-K 候选重排序，保持原始召回分与重排分以便分析。 |
| RAG 可靠性 | query rewrite、上下文去重、引用约束、无证据拒答。 | 回答必须引用检索到的片段；证据不足时明确拒答而非补全。 |
| 模型服务韧性 | 超时、重试、限额、缓存、成本统计与降级。 | 模型服务不可用时仍返回检索证据和明确提示，而不是让整个研究流程失败。 |
| 离线评测 | 评测集、Recall@K、MRR、nDCG、引用正确率、回答忠实度、拒答正确率与人工标注一致性。 | 围绕“生成式虚假信息检测与治理”等真实主题建立小型标注集和回归测试。 |

#### 推荐的实施顺序

1. 先选定一批真实问题及其相关论文/段落，建立 30–50 条小型检索评测集。
2. 记录当前摘要向量检索的 Recall@K、MRR、延迟和成本，作为基线。
3. 接入 PDF 全文和结构化元数据，先验证“证据可定位”是否改善。
4. 再逐项比较分块、混合检索、Embedding、重排序和 Prompt；一次只改变一个变量。
5. 将表现最好的配置固化为默认策略，并把关键样例纳入回归测试。

#### 最小交付与验收

- 对至少一个专题，用户可在详情页跳转到回答引用的论文和原文位置，并能查看该结果采用的数据与索引版本。
- 全文下载或解析失败时，论文保留来源与失败状态，摘要检索仍可用，用户不会把“无全文”误解为“无论文”。
- 仓库中存在可运行的评测脚本、版本化评测数据和结果报告。
- 每次检索策略改动都能回答：召回、排序、引用正确性、回答忠实度和拒答表现如何变化，延迟与成本增加多少，失败案例是什么。
- 明确区分“检索到了相关证据”“LLM 正确概括证据”“系统无证据时拒答”三类验收。
- 上游模型服务超时或不可用时，系统可保留检索结果与来源，并以可理解的方式说明生成回答暂不可用。

#### 这一阶段应能说明的能力

“构建论文全文 RAG 评测闭环，以 Recall@K/MRR 和人工案例评审驱动分块、混合检索与重排序优化，并通过页码级来源与拒答策略提升回答可核验性。”

### 阶段 5：V1.1 研究工作流与证据化 Idea

**目标：** 在 V1.0 的检索质量和工程基础稳定后，让用户不是“问完即走”，而是能把阅读、判断和研究假设留下来，形成 PaperMind 区别于普通 RAG Demo 的业务价值。

**实施手册：** [阶段 5：研究工作流与 Idea 卡片](./阶段5-研究工作流与Idea卡片实施手册.md)

#### 需要学习并落地的内容

| 主题 | 学习重点 | PaperMind 落地 |
| --- | --- | --- |
| 领域工作流设计 | 用户访谈、任务流、信息架构、可撤销操作。 | 围绕专题阅读、比较、记录与验证，而非泛化聊天功能堆叠。 |
| 论文比较 | 结构化提取、对比维度、证据引用。 | 并列比较研究问题、方法、数据、发现、局限和冲突点，并保留原文来源。 |
| 笔记与阅读状态 | 富文本或 Markdown、标签、引用关联、全文检索。 | 收藏、阅读进度、个人笔记与论文/片段关联。 |
| Idea 卡片 | 假设、支持证据、冲突证据、待验证问题、状态流转。 | 由用户确认后保存，绝不把模型生成内容直接当作可信结论。 |
| 可追溯生成 | 结构化输出校验、来源完整性、人工编辑。 | 每个 AI 辅助结论均能追溯到具体论文和片段。 |

#### 最小交付与验收

- 用户可选择多篇论文发起比较，查看结构化结果及每一项的来源。
- 用户可保存笔记和 Idea 卡片，并关联专题、论文和证据片段。
- Idea 卡片至少有“草稿、待验证、已验证/已放弃”状态，且支持人工修改。
- AI 生成的对比或 Idea 缺少证据时，应标记为“待核对”，而不是给出确定性结论。

#### 这一阶段应能说明的能力

“将 RAG 对话延伸为研究工作流：围绕论文比较、阅读笔记与可追溯 Idea 卡片设计领域实体和状态，使 AI 输出可由用户复核与持续迭代。”

## 五、版本策略与发布门槛

版本使用 `v0.y.z` / `v1.y.z` 形式。当前项目仍处于架构与产品验证期，从 `v0.1.0` 开始是合适的：`0` 主版本明确表示接口、数据模型与部署形态仍可能随验证调整；当 V1.0 业务闭环和发布门槛均通过后，再发布 `v1.0.0`。

| 版本 | 发布范围 | 发布门槛 |
| --- | --- | --- |
| `v0.1.0` | 当前后端 RAG MVP：关键词采集、摘要索引、检索、带来源问答与 Gradio 验收。 | 核心测试通过，README 能在本地复现当前流程。 |
| `v0.1.z` | 缺陷修复、测试补充、文档修正和不改变业务能力的维护。 | 不引入新的核心实体、接口语义或用户路径。 |
| `v0.2.0` | 最小持久化与正式前端。 | 阶段 1 验收通过。 |
| `v0.3.0` | 专题订阅、增量采集与后台任务。 | 阶段 2 验收通过。 |
| `v0.4.0` | 全文解析、可定位来源和检索评测闭环。 | 阶段 3 验收通过。 |
| `v0.5.0` | V1.0 发布候选：CI、部署、日志、备份恢复和完整验收。 | 阶段 4 验收通过，已知限制被记录。 |
| `v1.0.0` | 首个可稳定演示和复现部署的单用户产品版本。 | V1.0 边界全部满足，完成发布验收与演示脚本核对。 |
| `v1.1.0` | 论文比较、笔记和证据化 Idea 卡片。 | 阶段 5 验收通过，新增生成内容均可追溯与人工编辑。 |

发布规则：

- 每次 `v0.y.0` 或 `v1.y.0` 只交付一个可演示能力增量，并附对应的验收记录。
- `z` 只用于兼容性维护；若改变数据模型、API 语义或核心用户路径，应提升 `y` 并提供迁移、回滚和变更说明。
- 版本标签只在测试、迁移检查和核心人工路径验收通过后创建；不要用未验证功能抢占版本号。

## 七、每一阶段的工作模板

每准备做一个能力点，先填写以下内容，再开始实现。这样可以避免“看了很多资料、却没有形成项目产出”。

```markdown
### 能力点：<例如：论文全文混合检索>

**用户问题：** 用户在哪一步遇到什么困难？
**成功标准：** 用哪个可观察结果判断它解决了问题？
**最小方案：** 本次只做哪些接口、数据字段和页面？
**不做什么：** 哪些相邻功能明确延后？
**学习材料：** 只列解决当前实现所需的官方文档/论文。
**验证：** 单元测试、集成测试、人工用例或离线指标分别是什么？
**决策记录：** 选择方案的原因、替代方案和已知代价。
**版本与回滚：** 依赖哪些数据、模型或索引版本？失败后如何恢复？
```

完成后至少留下以下一种证据：代码与测试、接口示例、评测报告、架构决策记录或演示录屏。这样，学习过程会自然转化为项目的可讲述性。

## 八、阶段门槛与推荐推进方式

### 推荐顺序

1. 先完成阶段 1 的最小持久化、迁移和正式前端，让项目可被实际体验且不会因数据模型返工。
2. 紧接着完成阶段 2 的订阅、任务和数据质量，形成可持续的数据闭环。
3. 再投入阶段 3 的全文与评测；这是解决“当前 RAG 效果不够满意”的正确入口。
4. 检索基础稳定后完成阶段 4 的工程交付和 V1.0 发布，再开始 V1.1 工作流功能。
5. 阶段 5 的比较、笔记和 Idea 卡片必须建立在 V1.0 的可追溯证据和稳定发布基础上，避免在不可靠的检索上堆叠功能。

### 各阶段的停止条件

| 阶段 | 可以进入下一阶段的条件 |
| --- | --- |
| 阶段 1 | 用户能独立完成“进入专题 → 找到论文 → 检索/对话 → 查看来源”的页面路径；最小领域模型可通过迁移创建、升级与测试。 |
| 阶段 2 | 订阅与任务状态可追溯；增量采集可重复运行、可幂等重试，且长耗时工作不阻塞 API 请求。 |
| 阶段 3 | 有评测集和基线；全文证据可定位；数据、解析与索引版本可追溯；每项检索改动有指标或案例证据。 |
| 阶段 4 | 陌生环境能按文档运行和验收；有测试、日志、备份恢复、产品指标、演示与诚实的项目说明；满足 V1.0 发布门槛。 |
| 阶段 5 | AI 生成内容可回溯、可编辑、可标记验证状态，不会伪装成已证实的结论。 |

## 九、简历与面试的使用原则

项目成熟后，简历描述应使用“业务目标 + 技术方案 + 可验证结果”的结构。例如：

> 设计并实现 PaperMind 预印本研究情报应用：基于 FastAPI、PostgreSQL、ChromaDB 和 DeepSeek/SiliconFlow 构建论文采集、全文检索与证据化问答链路；通过离线检索评测优化召回与重排序，并以容器化和自动化测试保障多服务部署与复现。

其中“全文检索”“离线评测”“重排序”“容器化部署”等词必须在相应阶段真正完成后才写入。面试准备不应只背术语，而应能围绕以下问题给出自己项目中的取舍：

- 为什么当前先做专题与数据闭环，而不是先添加更多模型？
- 文档如何分块、为什么这样分、不同方案的指标和失败案例是什么？
- 如何保证一篇论文不会重复入库，后台任务失败如何恢复？
- LLM 回答依据在哪里，证据不足时系统如何处理？
- 为什么选择当前前端、向量库和部署结构，替代方案的代价是什么？
