> **目标：** AI 应用开发工程师（中厂/独角兽，AIGC 方向）
> **核心项目：** PaperMind（AI 论文知识库，RAG 全流程）
> **周期：** ~12 周（每周 10-20 小时）
> **起点对照：** 有 NLP 研究背景 + PyTorch 完整实验经验 + Flask/MySQL/Redis 接触过 + 日常 WSL 开发 + 爬虫经验

## 一、定位与策略

### 核心原则

| 原则             | 说明                                     |
| -------------- | -------------------------------------- |
| **一个项目覆盖核心技能** | 不做多个小项目。一个 RAG PaperMind贯穿所有技术点        |
| **金字塔式学习**     | 每个新技术：先花 30-60 分钟理解概念 → 写最小原型 → 再进项目集成 |
| **够用就好**       | 只学面试和工作前 3 个月会用到的东西，不追求体系完备            |
| **边准备边投**      | 第 6 周项目联调通过后就可以开始投简历，面试本身就是最好的学习驱动力    |

### 不做的事

- ❌ 不刷大量 LeetCode（中厂 AIGC 方向算法题要求不高，40-60 道高频 Medium 即可）
- ❌ 不系统学 React/Vue（AI 应用 Demo 用 Gradio 更高效）
- ❌ 不看大段视频教程（以文档 + 项目 + 搜索为主）
- ❌ 不背八股文（理解原理，用自己的话能说清楚即可）

### 金字塔学习法（本文档的核心组织方式）

每一项新技术，按三层结构推进：

```
第一层：概念（30-60 分钟）
  只看最核心的概念，不深挖细节
  目标：知道「这东西是什么、解决什么问题」

第二层：最小原型（30-60 分钟）
  写一个最简单的可运行代码
  目标：确认你理解了基本用法

第三层：项目集成
  放进PaperMind项目中落地
  目标：在真实场景中巩固，遇到问题知道查什么
```

### 第 3 周：采集模块（async + httpx + arXiv API）

#### async / await

```
第一层：概念（40 分钟）
  - 同步 vs 异步：等 I/O 时能不能干别的事
  - 协程是什么：单线程内协作式切换
  - async def / await / asyncio.run / asyncio.gather
  - async vs threading：单线程+协作 vs 多线程+抢占
  - 什么时候用 async：I/O 密集型（网络请求、文件读写）

  推荐阅读：
  - https://docs.python.org/3/library/asyncio-task.html 的 Coroutines 和 Tasks 两节
  - 搜索 "async vs threading python" 看一篇对比文章

第二层：最小原型（30 分钟）
  import asyncio
  import httpx

  async def fetch_one(url: str) -> int:
      async with httpx.AsyncClient() as client:
          resp = await client.get(url)
          return resp.status_code

  async def main():
      urls = ["https://httpbin.org/delay/1"] * 5
      results = await asyncio.gather(*[fetch_one(u) for u in urls])
      print(results)  # 应该约 1 秒完成，不是 5 秒

  asyncio.run(main())

第三层：项目集成
  collector.py 职责：
    - search_papers(keyword, max_results) → 论文列表
    - async batch_fetch(ids) → 并发抓取多篇详情
  arXiv API：http://export.arxiv.org/api/query?search_query=all:NLP&max_results=10
  解析 XML：用 feedparser 或 xml.etree
```

**设计考量：** 为什么用 async 而不是多线程？
- 爬虫是 I/O 密集型（等网络响应），不是 CPU 密集型
- async 更轻量，开 1000 个协程比开 1000 个线程省资源
- Python GIL 限制下，多线程对 I/O 密集型任务帮助不大

**替代方案对比：**
| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| async | 轻量、省内存 | 代码写法不同 | I/O 密集 |
| 多线程 | 写法简单 | GIL 限制、线程开销大 | CPU 不密集且不想改代码 |
| 多进程 | 真并行 | 内存开销大、进程间通信麻烦 | CPU 密集 |

### 第 4 周：索引模块（ChromaDB + Embedding）

#### 向量数据库 + Embedding

```
第一层：概念（45 分钟）
  - 什么是向量数据库：存向量 + 近似最近邻检索
  - ChromaDB vs MySQL：MySQL 存结构化数据，ChromaDB 存向量
  - Embedding：文本 → 固定长度向量（语义相近的文本向量也相近）
  - 分块（Chunking）为什么重要：LLM 上下文窗口有限、检索粒度控制

第二层：最小原型（40 分钟）
  pip install chromadb sentence-transformers

  import chromadb
  from chromadb.utils import embedding_functions

  client = chromadb.PersistentClient(path="./chroma_db")
  collection = client.create_collection(
      name="papers",
      embedding_function=embedding_functions.
          SentenceTransformerEmbeddingFunction(
              model_name="BAAI/bge-small-zh-v1.5"
          )
  )

  collection.add(
      documents=["第一篇论文的摘要", "第二篇论文的摘要"],
      metadatas=[{"arxiv_id": "2301.1"}, {"arxiv_id": "2301.2"}],
      ids=["paper_1", "paper_2"]
  )

  results = collection.query(query_texts=["机器学习"], n_results=2)
  print(results)

第三层：项目集成
  indexer.py：
    - 读取第 3 周采集的论文 JSON
    - 分块策略：按 256-512 tokens，重叠 32-64 tokens
    - 生成 Embedding 并存入 ChromaDB
    - 支持增量添加（新论文不覆盖旧的）

  retriever.py：
    - search(query, k=5) → 返回最相关的论文块

  pip install langchain-text-splitters（只取分块工具，不依赖全家桶）
  from langchain_text_splitters import RecursiveCharacterTextSplitter
```

**设计考量：** 为什么选 ChromaDB 而不是 Pinecone/Milvus？
- 本地可运行，零配置，适合个人项目
- 持久化存储（PersistentClient），进程重启数据不丢
- 如果项目要扩展到生产环境，Milvus 或 Pinecone 更合适

**设计考量：** 为什么分块选 512 tokens？
- 太小（128）：上下文不完整，语义碎片化
- 太大（2048）：检索粒度粗，可能混入无关信息
- 512 是常见的平衡点，但实际应根据数据特点调整

### 第 5 周：问答模块（LLM API + Prompt 工程）

#### LLM API + RAG Prompt

```
第一层：概念（30 分钟）
  - LLM API 基本格式：messages = [{role, content}]
  - 常用参数：temperature、max_tokens、top_p
  - RAG Prompt 模板：检索结果作为上下文注入 + 系统指令

第二层：最小原型（30 分钟，以 DeepSeek API 为例）
  from openai import OpenAI

  client = OpenAI(
      api_key="your-api-key",
      base_url="https://api.deepseek.com"
  )

  response = client.chat.completions.create(
      model="deepseek-chat",
      messages=[
          {"role": "system", "content": "你是一个PaperMind..."},
          {"role": "user", "content": "介绍一下这篇论文"}
      ]
  )
  print(response.choices[0].message.content)

第三层：项目集成
  qa_engine.py：
    - 接收用户问题
    - 调 retriever.search() 检索相关论文块
    - 构造 RAG Prompt（上下文 + 问题）
    - 调 LLM API 生成回答
    - 返回回答 + 引用的论文来源

  POST /ask → 请求：{"question": "...", "top_k": 5}
             响应：{"answer": "...", "sources": [...]}
```

**RAG Prompt 模板：**
```python
SYSTEM_PROMPT = """你是一个 AI 论文研究助手。
请基于提供的论文片段来回答问题。
如果提供的片段不足以回答问题，请如实说明。
引用来源时标注 [来源：论文标题]"""

def build_rag_prompt(question: str, contexts: list[dict]) -> str:
    context_str = "\n\n".join([
        f"[{i+1}] {ctx['text']}\n（来源：{ctx['title']}）"
        for i, ctx in enumerate(contexts)
    ])
    return f"""请基于以下论文内容回答问题：

{context_str}

问题：{question}"""
```

**设计考量：** 为什么用 DeepSeek/通义而不是直接调 OpenAI？
- 国内访问稳定
- 价格便宜（适合个人项目频繁调试）
- 中文支持好
- 如果项目要出海，再考虑 OpenAI/Claude

**设计考量：** 为什么 Prompt 里要加"如果片段不足以回答，请如实说明"？
- 防止 LLM hallucination（编造不存在的信息）
- 建立用户信任：承认不知道比瞎编好
- 这也是 RAG 系统的核心优势：回答可溯源

## 附录 B：常见坑与解决方案

| 预计遇到的坑 | 原因 | 解决方案 |
|---|---|---|
| ChromaDB 数据丢失 | 默认内存模式，进程结束数据就没了 | 用 `PersistentClient(path="./chroma_db")` |
| async 函数里调同步代码阻塞 | 在 async def 里用了 requests 而不是 httpx | 全部换成 httpx.AsyncClient |
| Docker build 很慢 | 每次改代码都重新装依赖 | 把 requirements.txt COPY 放前面，代码放后面 |
| LLM API 费用超标 | 频繁调试时调用了大量 API | 开发阶段用 mock 或本地小模型 |
| Gradio 页面加载模型很慢 | 模型在页面加载时初始化 | 用 Lazy Loading，第一次请求时才加载 |
| Embedding 模型下载慢 | 首次运行要下载几 GB 模型 | 提前下载好，或者换更小的模型 |
| 分块后语义不完整 | 在句子中间切断 | 用 RecursiveCharacterTextSplitter，设置合适的 separators |
| 检索结果不相关 | Embedding 模型和任务不匹配 | 换 domain-specific 的 Embedding 模型 |

## 附录 C：技术选型对比

### 后端框架

| 框架 | 优点 | 缺点 | 本项目选择 |
|---|---|---|---|
| FastAPI | 现代、async 原生、自动文档 | 学习曲线略高于 Flask | ✅ 选用 |
| Flask | 简单、社区大 | 不支持 async（要用第三方扩展） | ❌ 不选用 |
| Django | 功能全、生态完善 | 太重，不适合小型 API 服务 | ❌ 不选用 |

### 向量数据库

| 数据库 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| ChromaDB | 本地运行、零配置、易上手 | 性能不如专业向量库 | 个人项目、原型开发 |
| Milvus | 高性能、分布式 | 部署复杂 | 生产环境、大规模数据 |
| Pinecone | 托管服务、免运维 | 付费、数据隐私顾虑 | 快速上线、不想管基础设施 |
| Weaviate | 功能丰富、GraphQL 接口 | 学习成本高 | 需要复杂查询的场景 |

### LLM API

| 提供商 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| DeepSeek | 便宜、中文好、国内稳定 | 模型能力略逊于 GPT-4 | 个人项目、频繁调试 |
| 通义千问 | 国内访问快、中文好 | 价格中等 | 国内业务 |
| OpenAI | 能力最强、生态最完善 | 国内访问不稳定、贵 | 高质量要求、出海业务 |
| Claude | 长文本能力强 | 国内访问受限 | 需要处理长文档 |

### 前端工具

| 工具 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| Gradio | 极简、Python 原生、HuggingFace 生态 | 定制性有限 | AI Demo、快速原型 |
| Streamlit | 功能稍多、支持更多组件 | 比 Gradio 重一些 | 数据应用、复杂交互 |
| React | 灵活、生态完善 | 需要前端知识、开发慢 | 正式产品、复杂 UI |
| Vue | 简单易学、国内流行 | 需要前端知识 | 正式产品、国内团队 |
