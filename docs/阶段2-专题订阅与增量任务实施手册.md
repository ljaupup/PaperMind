# 阶段 2：专题订阅与增量任务实施手册

> **目标版本：** `v0.3.0`
>
> **阶段目标：** 将一次性、同步的论文采集升级为可调度、可追踪、可重试的增量任务系统。
>
> **完成后用户能做什么：** 为专题配置关键词和更新频率，手动触发或等待系统采集；查看任务进度、失败原因、新增论文数量和索引状态。

本阶段建立 PaperMind 的“持续数据闭环”。阶段 1 的专题是数据容器；阶段 2 让它主动获得新论文。长耗时操作不能继续在 FastAPI 请求进程中执行。

---

## 0. 先认识后台任务

浏览器请求应尽快返回；下载论文这类耗时操作则交给**后台任务**。任务队列像待办清单：API 创建一张任务单，worker 取走并执行，数据库保存任务状态。**幂等键**则让“同一件事重复点击”仍只产生一份有效任务。

下面的例子先不用 Celery，只展示状态为何要单独保存：

```python
"""演示一个最小任务状态机。"""

job = {"status": "pending"}


def start_job(item: dict[str, str]) -> None:
    """将待执行任务标记为运行中。

    :param item: 保存任务状态的可变字典。
    :raises ValueError: 任务不处于待执行状态时抛出。
    """
    if item["status"] != "pending":
        raise ValueError("任务不能重复启动")
    item["status"] = "running"
```

调用后 `job["status"]` 变为 `"running"`。本章再将这个字典替换为 `CollectionJob` 表，并由 Celery worker 完成真实采集。

## 0. 范围、技术选择与开始条件

### 固定技术选择

| 领域 | 选择 | 原因 |
| --- | --- | --- |
| Broker / Result Backend | Redis | 部署简单，适合任务消息、延迟任务和短期状态。 |
| Task Queue | Celery | Python 后端常见，具备 worker、retry、beat 调度与可观测性基础。 |
| 调度 | Celery Beat | 订阅更新频率应是任务调度，而不是在 Web 请求内轮询。 |
| 持久化真相来源 | PostgreSQL | Job 状态、统计和错误摘要必须落库，不能只依赖 Redis 结果过期时间。 |

在开始前确认阶段 1 已完成：专题有稳定 ID，`/api/v1/topics` 可用，Alembic 迁移已接管表结构。

### 本阶段不做什么

- 不做 PDF 全文下载和解析；只处理元数据与摘要增量。
- 不做分布式多地区调度、优先级队列或复杂工作流编排。
- 不在任务中直接生成 RAG 回答。
- 不把 Celery/Redis 当作唯一数据源；最终状态必须写回 PostgreSQL。

## 1. 先建模任务，而不是先写 Worker

### 1.1 新增资源

| 资源 | 核心字段 | 说明 |
| --- | --- | --- |
| `Subscription` | `id`、`topic_id`、`query`、`categories`、`max_results`、`schedule`、`is_enabled`、`last_collected_at` | 一个专题可有一条默认订阅；复杂多订阅留到确有需求时。 |
| `CollectionJob` | `id`、`topic_id`、`subscription_id`、`status`、`trigger_type`、`idempotency_key`、时间字段、统计、错误摘要 | 一次采集请求的持久化事实。 |
| `IndexJob` | `id`、`topic_id`、`status`、索引版本、时间字段、统计、错误摘要 | 本阶段可先建模；实际全文索引能力在阶段 3 扩展。 |

### 1.2 状态机

统一使用：

```text
pending → running → succeeded
                  ↘ failed
pending/running → cancelled（仅在真正实现取消时开放）
```

规则：

- `succeeded`、`failed`、`cancelled` 是终态，不能重新改回 `running`。
- 重试应创建新的 attempt 记录，或明确记录 `retry_count` 与每次错误；不要覆盖原失败原因。
- 同一专题、同一订阅、同一采集窗口只能存在一个活跃任务。
- `idempotency_key` 由 API 或应用服务生成；重复提交必须返回已有活跃 Job，而不是重复采集。

本节在阶段 1 的专题和 `/api/v1` 基础上继续。代码的重点是：API 只创建 Job，worker 执行采集，PostgreSQL 保存最终状态。

```python
from datetime import datetime
from enum import StrEnum
from uuid import UUID, uuid4

from pydantic import BaseModel, Field


class JobStatus(StrEnum):
    """定义采集任务的可用状态。"""

    PENDING = "pending"
    RUNNING = "running"
    SUCCEEDED = "succeeded"
    FAILED = "failed"


TERMINAL_STATUSES = {JobStatus.SUCCEEDED, JobStatus.FAILED}


class CollectionJob(BaseModel):
    """表示一次可追溯的论文采集任务。"""

    id: UUID = Field(default_factory=uuid4)
    topic_id: UUID
    subscription_id: UUID | None = None
    status: JobStatus = JobStatus.PENDING
    trigger_type: str = "manual"
    idempotency_key: str
    fetched_count: int = 0
    created_count: int = 0
    updated_count: int = 0
    failed_count: int = 0
    error_message: str | None = None
    created_at: datetime | None = None
    started_at: datetime | None = None
    finished_at: datetime | None = None

    def can_transition_to(self, target: JobStatus) -> bool:
        return (self.status, target) in {
            (JobStatus.PENDING, JobStatus.RUNNING),
            (JobStatus.RUNNING, JobStatus.SUCCEEDED),
            (JobStatus.RUNNING, JobStatus.FAILED),
        }
```

`StrEnum` 让状态既是枚举成员又可作为字符串保存；`Field(default_factory=uuid4)` 会在未传入 `id` 时生成 UUID。`trigger_type` 记录任务来源：`manual` 为用户触发、`scheduled` 为定时触发、`retry` 为重试触发。

创建 `app/domain/jobs/ports.py`：

```python
from typing import Protocol
from uuid import UUID

from app.domain.jobs.models import CollectionJob, JobStatus


class CollectionJobRepository(Protocol):
    """定义采集任务的持久化操作。"""

    def create_if_absent(self, job: CollectionJob) -> tuple[CollectionJob, bool]: ...
    def get(self, job_id: UUID) -> CollectionJob | None: ...
    def mark_running(self, job_id: UUID) -> CollectionJob | None: ...
    def finish_success(self, job_id: UUID, *, fetched: int, created: int, updated: int) -> CollectionJob: ...
    def finish_failure(self, job_id: UUID, message: str) -> CollectionJob: ...


class TaskDispatcher(Protocol):
    """定义将采集任务交给后台执行器的操作。"""

    def dispatch_collection(self, job_id: UUID) -> None: ...
```

应用服务必须完成：

建议先使用 `FakeJobRepository` 与 `FakeTaskDispatcher` 写测试。重点断言：重复请求不会产生两个活跃 Job；投递失败时 Job 不应停留在无解释的 `pending` 状态。

#### 应用服务：创建 Job 而不是直接采集

创建 `app/application/collection_job_service.py`：

```python
from datetime import UTC, datetime
from uuid import UUID

from app.domain.jobs.models import CollectionJob
from app.domain.jobs.ports import CollectionJobRepository, TaskDispatcher


class CollectionJobService:
    """创建任务并通过调度器提交执行。"""

    def __init__(self, repository: CollectionJobRepository, dispatcher: TaskDispatcher) -> None:
        self.repository = repository
        self.dispatcher = dispatcher

    def create(self, topic_id: UUID, *, idempotency_key: str, trigger_type: str = "manual") -> CollectionJob:
        candidate = CollectionJob(
            topic_id=topic_id,
            idempotency_key=idempotency_key,
            trigger_type=trigger_type,
            created_at=datetime.now(UTC),
        )
        job, created = self.repository.create_if_absent(candidate)
        if created:
            try:
                self.dispatcher.dispatch_collection(job.id)
            except Exception as exc:
                self.repository.finish_failure(job.id, f"unable to enqueue task: {exc}")
                raise
        return job

    def get_or_raise(self, job_id: UUID) -> CollectionJob:
        job = self.repository.get(job_id)
        if job is None:
            raise LookupError("collection job not found")
        return job
```

对应测试应验证同一幂等键第二次调用不会再次调用 `dispatch_collection`：

### 2.2 PostgreSQL 与 Alembic

创建 Subscription、CollectionJob、IndexJob 的 ORM 和迁移。最低数据库约束：

- `subscriptions.topic_id` 外键指向专题。
- `collection_jobs.topic_id`、`subscription_id` 外键与必要索引。
- `status`、`created_at`、`started_at`、`finished_at` 建组合查询索引。
- `idempotency_key` 在合理范围内唯一，避免双击按钮或网络重试触发重复采集。
- 统计字段至少包含 `fetched_count`、`created_count`、`updated_count`、`failed_count`。

迁移必须在空库和已有阶段 1 数据库各验证一次。

生成迁移前先在 `migrations/env.py` 导入 `job_models`，然后执行：

前置条件：Subscription、CollectionJob 与 IndexJob 的 ORM 模型已加入 `Base.metadata`，并已在可丢弃数据库审阅过生成迁移。满足后生成并应用迁移：


Repository 的关键是原子状态切换。以 `mark_running` 为例：

```python
from datetime import UTC, datetime
from sqlalchemy import update

def mark_running(self, job_id):
    with self.session_factory() as session:
        statement = (
            update(CollectionJobRecord)
            .where(CollectionJobRecord.id == job_id)
            .where(CollectionJobRecord.status == "pending")
            .values(status="running", started_at=datetime.now(UTC))
            .returning(CollectionJobRecord)
        )
        record = session.execute(statement).scalar_one_or_none()
        session.commit()
        return None if record is None else to_domain(record)
```

只有成功从 `pending` 更新为 `running` 的 worker 才继续采集；其他重复投递的 worker 应直接退出。

### 2.3 Celery 与 Redis

在 Docker Compose 中加入 Redis、worker、beat 服务；依赖安装与 Redis 启动命令放在下方的实现步骤中执行。

建议目录：

```text
app/infrastructure/tasks/
├── celery_app.py
├── dispatcher.py
├── collection_tasks.py
└── schedules.py
```

`collection_tasks.py` 的工作顺序：

```text
读取 Job
→ 原子地切换 pending 到 running
→ 调用 arXiv Collector
→ 按 arXiv ID 去重并保存 Paper
→ 关联 TopicPaper
→ 写入统计
→ 切换 succeeded
```

异常处理：

- 可重试错误：临时网络失败、上游 `5xx`、可恢复连接错误；使用指数退避。
- 不应立即重试：参数错误、专题/订阅已删除、明显的 `429` 限流；记录错误并按策略等待下个窗口。
- 无论成功或失败，写入 `finished_at` 和可读错误摘要。

创建 `app/infrastructure/tasks/celery_app.py`：

```python
from celery import Celery

from app.core.config import Settings

settings = Settings()
celery_app = Celery(
    "papermind",
    broker=settings.redis_url,
    backend=settings.redis_url,
    include=["app.infrastructure.tasks.collection_tasks"],
)
celery_app.conf.update(
    task_track_started=True,
    task_acks_late=True,
    worker_prefetch_multiplier=1,
    task_time_limit=15 * 60,

```

`Celery()` 的 `broker` 指向任务队列，`backend` 保存 Celery 的短期执行结果；本项目的业务状态仍以 PostgreSQL 为准。`include` 指定 worker 启动时要加载的任务模块。`task_acks_late=True` 表示任务完成后才确认，worker 意外退出时队列可重新投递；这也要求任务本身具备幂等性。

因此需要在 `Settings` 中新增：


创建 `app/infrastructure/tasks/collection_tasks.py`。Celery Task 是基础设施适配器，不能把业务规则全部写在这里；它获取容器中的应用服务：

```python
from uuid import UUID

from app.core.container import AppContainer
from app.infrastructure.tasks.celery_app import celery_app


@celery_app.task(bind=True, autoretry_for=(ConnectionError,), retry_backoff=True, max_retries=3)
def run_collection_job(self, job_id: str) -> None:
    container = AppContainer()
    service = container.collection_execution_service
    service.execute(UUID(job_id))
```

创建 `app/application/collection_execution_service.py`。`execute()` 只做 `mark_running`、读取专题、调用采集器、保存关联和写终态；具体 SQL 仍封装在 Repository 内：

```python
import asyncio
from uuid import UUID

from app.domain.jobs.ports import CollectionJobRepository
from app.domain.topics.ports import TopicRepository


class CollectionExecutionService:
    """执行采集、关联论文并写回任务终态。"""

    def __init__(self, jobs: CollectionJobRepository, topics: TopicRepository, collector, paper_service) -> None:
        self.jobs = jobs
        self.topics = topics
        self.collector = collector
        self.paper_service = paper_service

    def execute(self, job_id: UUID) -> None:
        job = self.jobs.mark_running(job_id)
        if job is None:
            return

        try:
            topic = self.topics.get(job.topic_id)
            if topic is None:
                raise LookupError(f"topic not found: {job.topic_id}")

            papers = asyncio.run(self.collector.collect(keywords=topic.keywords))
            created = 0
            updated = 0
            for paper in papers:
                was_created = self.paper_service.upsert_and_link_to_topic(paper, topic.id)
                created += int(was_created)
                updated += int(not was_created)

            self.jobs.finish_success(
                job.id,
                fetched=len(papers),
                created=created,
                updated=updated,
            )
        except Exception as exc:
            self.jobs.finish_failure(job.id, str(exc))
            raise
```

`mark_running()` 必须是原子更新；返回空值表示任务已被另一 worker 取得或已经结束。此处复用现有 arXiv 适配器，并只在同步 Celery task 边界调用一次 `asyncio.run()`。捕获异常后先写入失败状态再重新抛出，使 Celery 按 task 的重试策略记录并处理异常。

在 `AppContainer` 中将 `collection_job_service`、`collection_execution_service` 和 `CeleryTaskDispatcher` 组装到一起。`PaperService.upsert_and_link_to_topic()` 必须在一个数据库事务中完成“按 arXiv ID 幂等 upsert 论文”和“按 `(topic_id, paper_id)` 幂等建立关联”。

前置条件：Redis 已通过 `PONG` 验证，任务表迁移已应用，`Settings.redis_url`、Celery app、collection task 和容器装配均已实现。然后在两个独立终端启动：

```bash

```

### 2.4 调度与增量语义

不要每次订阅运行都重新请求相同的全部论文。增量规则应明确：

1. 使用 arXiv ID 作为论文去重主键。
2. 记录订阅上次成功采集时间和最后处理的提交时间。
3. 请求结果按提交时间排序；允许少量重叠窗口，依赖去重抵抗延迟。
4. 只把新增或更新的论文关联到专题。
5. 调度器只负责创建 Job；真实采集仍由 worker 完成。

Celery Beat 的周期规则来源于 Subscription。早期可以每小时扫描一次启用订阅并创建任务，不必动态为每条订阅生成独立 cron 条目。

#### `202 Accepted` Router

创建 `app/schemas/jobs.py`：

```python
from datetime import datetime
from uuid import UUID

from pydantic import BaseModel


class CreateCollectionJobRequest(BaseModel):
    """校验创建采集任务的 HTTP 请求。"""

    idempotency_key: str | None = None


class CollectionJobResponse(BaseModel):
    """定义采集任务的 HTTP 响应。"""

    id: UUID
    topic_id: UUID
    status: str
    fetched_count: int
    created_count: int
    updated_count: int
    error_message: str | None
    created_at: datetime | None
    started_at: datetime | None
    finished_at: datetime | None

    @classmethod
    def from_domain(cls, job) -> "CollectionJobResponse":
        return cls(**job.model_dump())
```

创建 `app/api/v1/jobs.py`：

```python
from uuid import UUID

from fastapi import APIRouter, Depends, Header, HTTPException, status

from app.api.dependencies import get_collection_job_service
from app.application.collection_job_service import CollectionJobService
from app.schemas.jobs import CollectionJobResponse, CreateCollectionJobRequest

router = APIRouter(tags=["collection-jobs"])


@router.post(
    "/topics/{topic_id}/collection-jobs",
    response_model=CollectionJobResponse,
    status_code=status.HTTP_202_ACCEPTED,
)
def create_collection_job(
    topic_id: UUID,
    request: CreateCollectionJobRequest,
    idempotency_key: str | None = Header(default=None, alias="Idempotency-Key"),
    service: CollectionJobService = Depends(get_collection_job_service),
) -> CollectionJobResponse:
    key = idempotency_key or request.idempotency_key or f"manual:{topic_id}"
    job = service.create(topic_id, idempotency_key=key)
    return CollectionJobResponse.from_domain(job)


@router.get("/collection-jobs/{job_id}", response_model=CollectionJobResponse)
def get_collection_job(job_id: UUID, service: CollectionJobService = Depends(get_collection_job_service)):
    try:
        return CollectionJobResponse.from_domain(service.get_or_raise(job_id))
    except LookupError as exc:
        raise HTTPException(status_code=404, detail="collection job not found") from exc
```

### 3.3 `v0.3.0` 完成定义

- 采集、索引不在 Web 请求中执行。
- 所有 Job 状态与统计在 PostgreSQL 中可追溯。
- Redis/Celery 重启后，已完成 Job 历史仍可查询。
- 重复采集不会造成论文或专题关联重复。
- 前端能展示任务状态与失败原因。
- 新增迁移、任务和 API 测试全部通过。
- IndexJob 的状态、API 契约与阶段 3 所需的派发边界均已完成。

### 3.4 常见错误

| 问题 | 原因 | 处理 |
| --- | --- | --- |
| API 一直等待 arXiv 返回 | 在 Router 或 Service 中直接执行采集 | Router 只创建 Job，Worker 执行真实 I/O。 |
| Redis 中有结果但数据库无 Job | 把队列当作业务数据库 | PostgreSQL 是状态真相来源。 |
| 重试产生重复论文 | 缺少 arXiv ID 与 TopicPaper 唯一约束 | 用数据库约束兜底，应用层做幂等。 |
| Job 永远 `running` | Worker 崩溃未写终态 | 引入超时、心跳/过期检查和失败恢复策略。 |
| Beat 重复创建任务 | 调度扫描没有幂等窗口 | 创建 Job 时使用订阅+时间窗口的幂等键。 |

### 推荐提交边界

1. `feat: 增加订阅与任务领域模型`
2. `feat: 接入 Redis Celery 采集任务`
3. `feat: 提供任务 RESTful API 与前端状态展示`
4. `test: 覆盖任务状态机与幂等采集`

阶段结束后，你应能解释：为什么长任务必须异步化、为什么 Job 要落库、如何做到幂等和重试，以及 `202 Accepted` 与 `201 Created` 的区别。
