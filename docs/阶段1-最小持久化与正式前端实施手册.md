# 阶段 1：最小持久化与正式前端实施手册

> **目标版本：** `v0.2.0`
>
> **阶段目标：** 将当前“可调用的后端 RAG MVP”升级为有稳定领域数据、有 RESTful 专题 API、可在浏览器连续操作的单用户应用。
>
> **完成后用户能做什么：** 创建研究专题，查看专题中的论文，收藏论文，在专题范围内检索和问答，并在页面中核对回答来源。

本手册是 [学习路线](./学习路线.md) 的阶段 1 展开说明。它不重做已有的 arXiv 采集、摘要索引、Chroma 检索和 RAG 链路；它要解决的是“这些能力如何变成稳定产品”的问题。

## 开始前：从原有 `app/` 到当前分层结构

在开始阶段 1 前，先把一次已经完成的后端整理看作“交接”，而不是新的功能任务。原来的 MVP 把路由、编排、模型服务和数据库代码放在同一个 `app/` 目录中；这在功能很少时容易找到文件，但当增加专题、任务、全文和用户数据后，修改一个功能容易牵连多个职责。

**分层**就是按职责给代码划定边界：API 层处理 HTTP 请求，应用层组织一个用例，领域层表达业务规则与接口约定，基础设施层实现 PostgreSQL、arXiv、Chroma 和模型服务。这样更换底层技术时，业务规则不需要随之散落修改。

先看一个与 PaperMind 无关、可以直接运行的最小例子：

```python
class MemoryStore:
    """在内存中保存文本。"""

    def __init__(self) -> None:
        self.items: list[str] = []

    def save(self, text: str) -> None:
        """保存一条文本。"""
        self.items.append(text)


def create_note(store: MemoryStore, text: str) -> None:
    """创建并保存一条笔记。"""
    store.save(text.strip())


store = MemoryStore()
create_note(store, "  first note  ")
assert store.items == ["first note"]
```

这里 `create_note()` 只表达“创建笔记”这个用例；`MemoryStore` 负责“如何保存”。PaperMind 中的 `application` 与 `infrastructure` 也采用相同分工，只是将内存列表替换为可切换的 PostgreSQL、Chroma、arXiv 或模型服务适配器。

### 原文件与新位置的对应关系

下表是学习时最常用的定位表。箭头表示代码职责的去向，不代表每个旧文件都被逐字拆分到唯一的新文件。

| 原 MVP 文件 | 当前位置 | 为什么这样迁移 |
| --- | --- | --- |
| `app/main.py` | `app/main.py`、`app/api/router.py`、`app/api/routers/` | `main.py` 只创建应用并注册 Router；健康检查、论文采集、索引检索和问答分别成为小 Router。 |
| `app/config.py` | `app/core/config.py` | 环境变量与运行配置属于所有层都可读取的核心配置。 |
| `app/schemas.py` | `app/schemas/papers.py`、`knowledge.py`、`conversations.py` | 按 API 资源拆分请求/响应 DTO，避免一个文件同时承载所有接口格式。 |
| `app/collector.py` | `app/infrastructure/arxiv/collector.py` | arXiv SDK 和第三方返回对象是外部技术细节。 |
| `app/storage.py` | `app/infrastructure/persistence/repositories.py` | 本地 JSON Repository 与存储实现选择移入持久化适配器。 |
| `app/db/database.py`、`models.py`、`repositories.py` | `app/infrastructure/persistence/database.py`、`models.py`、`postgres_repository.py` | 数据库连接、ORM 表模型和 PostgreSQL 查询均属于基础设施。原 `app/db/` 不再保留源代码。 |
| `app/chunker.py` | `app/domain/knowledge/chunking.py` | 分块策略决定知识单元的业务语义，不能依赖 Chroma 或 PDF SDK。 |
| `app/embeddings.py`、`app/llm.py` | `app/infrastructure/ai/embeddings.py`、`llm.py` | SiliconFlow / DeepSeek 等 OpenAI 兼容调用是可替换的外部适配器。 |
| `app/vector_store.py` | `app/infrastructure/vector/chroma.py` | Chroma 的 collection、upsert 和 query API 与业务检索规则分离。 |
| `app/indexer.py`、`app/retriever.py`、`app/rag.py`、`app/qa_engine.py` | `app/application/indexing_service.py`、`retrieval_service.py`、`rag_service.py`，并复用 AI 适配器 | 原先分散或重叠的编排职责集中为“建立索引、检索、生成回答”三个用例；`qa_engine.py` 不再单独存在。 |

`app/domain/models.py` 与 `app/domain/ports.py` 是这次新增的关键边界：前者放 `Paper`、`Answer` 等不依赖数据库或 HTTP 的业务数据；后者以 `Protocol` 声明 `PaperRepository`、`EmbeddingClient`、`LLMClient`、`VectorStore` 等“应用需要什么能力”。基础设施层再提供这些能力的真实实现。

### 当前一次请求如何流动

```text
HTTP 请求
  → api/router.py 与 api/routers/       # 匹配路径、校验 HTTP 输入
  → api/dependencies.py                 # 从容器取得所需应用服务
  → application/*_service.py            # 编排“采集、索引、检索、问答”用例
  → domain/models.py 与 ports.py         # 业务数据和抽象能力约定
  → infrastructure/*                    # PostgreSQL、Chroma、arXiv、Embedding、LLM 的真实调用
```

`app/core/container.py` 是**组装根**：它读取 `Settings`，创建 Repository、向量库和模型客户端，再把它们注入应用服务。业务代码不应在 Router 内直接 `new` Chroma 客户端，也不应在领域模型内读取 `.env`；这样单元测试才能用 Fake 实现替代外部服务。

阶段 1 新增 `Topic` 时，沿用这一方向：先在 `domain` 定义专题及其端口，再在 `application` 编排用例，随后在 `infrastructure/persistence` 落地 ORM、迁移和 Repository，最后从 `api` 暴露 RESTful 资源。下文会按此顺序实施。

## 0. 先认识本阶段的三个概念

**持久化**是让数据在程序或浏览器刷新后仍能找回；数据库表保存数据，API 让前端能够读取和修改它。**资源**是可独立识别的业务对象，例如一个专题；RESTful API 用 URL 表示资源、用 HTTP 方法表示动作。**迁移**是数据库结构的版本记录，使新机器能按相同步骤创建同样的表。

先看与项目无关的最小例子：

```python
"""演示内存中的专题资源。"""

topics: list[dict[str, str]] = []


def create_topic(name: str) -> dict[str, str]:
    """创建一个专题字典。

    :return: 包含新专题标识和名称的字典。
    """
    topic = {"id": str(len(topics) + 1), "name": name}
    topics.append(topic)
    return topic
```

调用 `create_topic("RAG")` 会返回 `{"id": "1", "name": "RAG"}`。阶段 1 会把这个仅存于内存的字典，替换为 `Topic` 领域模型、PostgreSQL 表和 `/api/v1/topics` 接口。

## 0. 如何使用本手册

每完成一个小节都应执行对应验证，再进入下一节。不要先搭完整前端、再补数据模型；也不要为了“RESTful”只改接口路径而没有真实资源和状态。

本阶段固定采用以下技术选择：

| 领域 | 选择 | 原因 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 与后端独立部署，学习成本低，适合单页研究工作区。 |
| 数据迁移 | Alembic | 现有 SQLAlchemy 模型需要可版本化升级，`create_all()` 不适合后续演进。 |
| 后端 API | FastAPI `/api/v1` | 以专题、论文等真实资源为中心，并保留旧 `v0.1` 接口兼容期。 |
| 用户边界 | 单用户工作区 | 暂不做登录、多租户、RBAC；不把认证复杂度带入第一个产品版本。 |

## 1. 先定义产品资源与验收

### 1.1 用户路径

```text
进入专题列表
  → 创建“生成式虚假信息检测与治理”专题
  → 进入专题详情
  → 查看关联论文、筛选并收藏论文
  → 在专题范围内检索或提问
  → 展开回答来源，打开 arXiv 页面或 PDF 链接
```

### 1.2 本阶段领域模型

本阶段只新增三个概念：

| 模型 | 必要字段 | 业务规则 |
| --- | --- | --- |
| `Workspace` | `id`、`name`、创建时间 | 单用户固定工作区即可；为未来用户归属预留 `workspace_id`。 |
| `Topic` | `id`、`workspace_id`、`name`、`description`、`keywords`、`categories`、创建/更新时间 | 同一工作区内专题名称唯一；关键词不能为空。 |
| `TopicPaper` | `topic_id`、`paper_id`、`is_favorite`、创建时间 | 同一专题不能重复关联同一论文；收藏是专题内状态，而不是论文全局状态。 |

现有 `Paper` 仍是全局论文实体。不要在 `papers` 表中加入 `topic_id`，否则一篇论文无法属于多个专题。

### 1.3 RESTful API 契约

从本阶段开始新增接口统一使用 `/api/v1`。旧 `/papers/collect`、`/search`、`/ask` 暂保留为 `v0.1` 兼容入口，前端只调用新接口。

| 方法 | 路径 | 含义 | 成功响应 |
| --- | --- | --- | --- |
| `POST` | `/api/v1/topics` | 创建专题 | `201 Created` + `TopicResponse` |
| `GET` | `/api/v1/topics` | 分页查询专题 | `200 OK` |
| `GET` | `/api/v1/topics/{topic_id}` | 查询专题详情 | `200 OK` |
| `PATCH` | `/api/v1/topics/{topic_id}` | 局部更新专题 | `200 OK` |
| `DELETE` | `/api/v1/topics/{topic_id}` | 删除专题及其关联关系 | `204 No Content` |
| `GET` | `/api/v1/topics/{topic_id}/papers` | 查询专题论文 | `200 OK` |
| `POST` | `/api/v1/topics/{topic_id}/papers` | 关联已有论文 | `201 Created` |
| `PATCH` | `/api/v1/topics/{topic_id}/papers/{paper_id}` | 更新专题内收藏状态 | `200 OK` |

统一约定：

- 列表接口使用 `cursor` 与 `limit`，不要从第一天设计 offset 深分页。
- 所有不存在的资源返回 `404`；违反唯一约束返回 `409`；请求字段不合法由 FastAPI 返回 `422`。
- 自定义业务错误使用 `application/problem+json` 风格，至少包含 `type`、`title`、`status`、`detail`、`instance`。
- 只允许 `PATCH` 修改部分字段；不实现无意义的全量 `PUT`。

## 2. 后端实施顺序

### 2.1 安装并初始化 Alembic

先阅读 Alembic 的迁移概念：ORM 模型是“当前代码期望的结构”，迁移文件是“数据库从旧结构演进到新结构的历史”。不要继续使用 `Base.metadata.create_all()` 管理生产数据表。

需要完成的配置：

1. 在 `alembic.ini` 或迁移环境中读取 `Settings().postgres_url`，不把数据库密码写入迁移脚本。
2. 在 `migrations/env.py` 导入 SQLAlchemy `Base.metadata`，使 `--autogenerate` 能发现模型变化。
3. 为当前 `papers` 表建立初始迁移；不要假设已有开发数据库可以跳过这一步。
4. 在空数据库执行 `upgrade head`，再执行 `downgrade -1` 与 `upgrade head` 验证可回滚。

建议先写一个迁移检查测试或本地脚本：新建临时数据库后执行升级，确认 `papers`、`workspaces`、`topics`、`topic_papers` 表均存在。

本节按实际创建顺序给出代码。不要一次性复制所有内容；每完成一个小节先运行对应测试。以下示例以当前重构后的 `app/` 分层为基础，假设 PostgreSQL 是默认存储。

#### 加入 Alembic

前置条件：当前分支尚未存在 `migrations/`，并且已完成专题 ORM 模型的设计；`alembic init` 只能初始化一次。满足后安装并初始化：

```bash
uv add alembic
uv run alembic init migrations
```

在 `migrations/env.py` 中，不要写死数据库 URL。保留 `alembic init` 生成的其他模板代码，并将核心配置和 ORM 元数据接入 Alembic：

```python
from app.core.config import Settings
from app.infrastructure.persistence.models import Base
from app.infrastructure.persistence import topic_models as _topic_models

config.set_main_option("sqlalchemy.url", Settings().postgres_url)
target_metadata = Base.metadata
```

`topic_models` 的导入不能省略：Alembic 只有在模块被导入时，才能在 `Base.metadata` 中发现专题相关的表。

`config.set_main_option()` 是 Alembic 配置对象的方法，用来设置本次迁移使用的连接地址；`target_metadata` 是 SQLAlchemy 收集到的表结构，`--autogenerate` 据此生成候选迁移。生成的迁移不是自动可信代码，仍要审阅。

前置条件：`Base.metadata` 已导入现有 `PaperRecord` 与本阶段新增的 Workspace、Topic、TopicPaper ORM 模型，且 PostgreSQL 已启动并可连接。满足后创建初始迁移：

```bash
uv run alembic revision --autogenerate -m "add topics and topic papers"
uv run alembic upgrade head
```

生成后必须人工审阅 `migrations/versions/*.py`：确认迁移只创建本阶段需要的表、约束和索引，没有误删已有 `papers` 表。

```python
from typing import Protocol
from uuid import UUID

from app.domain.models import Paper
from app.domain.topics.models import Topic, TopicPaper


class TopicAlreadyExistsError(Exception):
    """表示同一工作区中的专题名称重复。"""

    pass


class TopicNotFoundError(Exception):
    """表示请求的专题不存在。"""

    pass


class TopicPaperAlreadyExistsError(Exception):
    """表示论文已关联到该专题。"""

    pass


class TopicRepository(Protocol):
    """定义专题数据访问所需的最小操作。"""

    def create(self, topic: Topic) -> Topic: ...
    def get(self, topic_id: UUID) -> Topic | None: ...
    def list(self, *, limit: int, cursor: UUID | None = None) -> list[Topic]: ...
    def update(self, topic_id: UUID, **values: object) -> Topic | None: ...
    def delete(self, topic_id: UUID) -> bool: ...
    def add_paper(self, link: TopicPaper) -> TopicPaper: ...
    def set_favorite(self, topic_id: UUID, paper_id: str, is_favorite: bool) -> TopicPaper | None: ...
    def list_papers(self, topic_id: UUID, *, limit: int) -> list[Paper]: ...
```

创建 `app/application/topic_service.py`，使上述测试通过：

```python
from uuid import UUID

from app.domain.topics.models import Topic
from app.domain.topics.ports import TopicNotFoundError, TopicRepository


class TopicService:
    """协调专题的创建、查询与输入规范化。"""

    def __init__(self, repository: TopicRepository, workspace_id: UUID) -> None:
        self.repository = repository
        self.workspace_id = workspace_id

    @staticmethod
    def _normalize_items(values: list[str]) -> list[str]:
        result: list[str] = []
        for value in values:
            normalized = value.strip().lower()
            if normalized and normalized not in result:
                result.append(normalized)
        return result

    def create(
        self,
        name: str,
        description: str,
        keywords: list[str],
        categories: list[str],
    ) -> Topic:
        normalized_keywords = self._normalize_items(keywords)
        if not normalized_keywords:
            raise ValueError("at least one keyword is required")
        topic = Topic(
            workspace_id=self.workspace_id,
            name=name.strip(),
            description=description.strip(),
            keywords=normalized_keywords,
            categories=self._normalize_items(categories),
        )
        return self.repository.create(topic)

    def get_or_raise(self, topic_id: UUID) -> Topic:
        topic = self.repository.get(topic_id)
        if topic is None:
            raise TopicNotFoundError(str(topic_id))
        return topic

    def list(self, limit: int, cursor: UUID | None = None) -> list[Topic]:
        return self.repository.list(limit=limit, cursor=cursor)
```

### 2.3 实现 PostgreSQL 持久化和迁移

只在领域端口和服务测试稳定后创建：

```text
app/infrastructure/persistence/models/topic.py
app/infrastructure/persistence/topic_repository.py
migrations/versions/<revision>_add_topics.py
```

实现细节：

- `topics.workspace_id + topics.name` 建唯一约束。
- `topic_papers.topic_id + topic_papers.paper_id` 建联合唯一约束。
- 查询专题论文时按关联创建时间或论文发布时间明确排序，不能依赖数据库默认顺序。
- 删除专题仅删除专题及其关联记录，不删除全局 `papers` 数据。
- 数据库异常在 Repository 层转换为可理解的领域异常，Router 不直接捕获 SQLAlchemy 异常。

完成后将真实 `TopicRepository` 加入 `app/core/container.py`；容器是唯一知道 PostgreSQL 实现的地方。

创建 `app/infrastructure/persistence/topic_repository.py`：

```python
from uuid import UUID

from sqlalchemy import select
from sqlalchemy.exc import IntegrityError
from sqlalchemy.orm import Session

from app.domain.topics.models import Topic
from app.domain.topics.ports import TopicAlreadyExistsError
from app.infrastructure.persistence.topic_models import TopicRecord


def to_topic(record: TopicRecord) -> Topic:
    """将 ORM 记录转换为领域模型。

    :param record: 从数据库读取的专题记录。
    :return: 不依赖 SQLAlchemy 的专题领域对象。
    """

    return Topic(
        id=record.id,
        workspace_id=record.workspace_id,
        name=record.name,
        description=record.description,
        keywords=list(record.keywords),
        categories=list(record.categories),
        created_at=record.created_at,
        updated_at=record.updated_at,
    )


class PostgresTopicRepository:
    """使用 PostgreSQL 保存和读取专题。"""

    def __init__(self, session_factory) -> None:
        self.session_factory = session_factory

    def create(self, topic: Topic) -> Topic:
        record = TopicRecord(**topic.model_dump())
        try:
            with self.session_factory() as session:
                session.add(record)
                session.commit()
                session.refresh(record)
                return to_topic(record)
        except IntegrityError as exc:
            raise TopicAlreadyExistsError(topic.name) from exc

    def get(self, topic_id: UUID) -> Topic | None:
        with self.session_factory() as session:
            record = session.get(TopicRecord, topic_id)
            return None if record is None else to_topic(record)

    def list(self, *, limit: int, cursor: UUID | None = None) -> list[Topic]:
        with self.session_factory() as session:
            statement = select(TopicRecord).order_by(TopicRecord.created_at.desc()).limit(limit)
            records = session.scalars(statement).all()
            return [to_topic(record) for record in records]
```

上例刻意没有“猜测” cursor 的排序逻辑。真正实现 cursor 前，应先确定使用 `(created_at, id)` 还是单调 ID，并为此写集成测试；不要提交看似分页、实际会漏数据的代码。

from fastapi import APIRouter, Depends, HTTPException, status

from app.api.dependencies import get_topic_service
from app.application.topic_service import TopicService
from app.domain.topics.ports import TopicAlreadyExistsError, TopicNotFoundError
from app.schemas.topics import CreateTopicRequest, TopicListResponse, TopicResponse


router = APIRouter(prefix="/topics", tags=["topics"])


@router.post("", response_model=TopicResponse, status_code=status.HTTP_201_CREATED)
def create_topic(
    request: CreateTopicRequest,
    service: TopicService = Depends(get_topic_service),
) -> TopicResponse:
    try:
        topic = service.create(**request.model_dump())
    except TopicAlreadyExistsError as exc:
        raise HTTPException(status_code=409, detail="topic name already exists") from exc
    return TopicResponse.from_domain(topic)


@router.get("", response_model=TopicListResponse)
def list_topics(
    limit: int = 20,
    service: TopicService = Depends(get_topic_service),
) -> TopicListResponse:
    topics = service.list(limit=min(max(limit, 1), 100))
    return TopicListResponse(items=[TopicResponse.from_domain(topic) for topic in topics])


@router.get("/{topic_id}", response_model=TopicResponse)
def get_topic(topic_id, service: TopicService = Depends(get_topic_service)) -> TopicResponse:
    """按标识查询一个专题。

    :param topic_id: 要查询的专题标识。
    :param service: 由 FastAPI 注入的专题应用服务。
    :return: 可发送给客户端的专题响应。
    """

    try:
        return TopicResponse.from_domain(service.get_or_raise(topic_id))
    except TopicNotFoundError as exc:
        raise HTTPException(status_code=404, detail="topic not found") from exc

```tsx
import { FormEvent, useEffect, useState } from "react";
import { createTopic, listTopics, Topic } from "../api/topics";

export function TopicsPage() {
  const [topics, setTopics] = useState<Topic[]>([]);
  const [name, setName] = useState("");
  const [keywords, setKeywords] = useState("");
  const [error, setError] = useState("");
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    listTopics().then(setTopics).catch((e) => setError(e.message)).finally(() => setLoading(false));
  }, []);

  async function onSubmit(event: FormEvent) {
    event.preventDefault();
    setError("");
    try {
      const topic = await createTopic({
        name,
        description: "",
        keywords: keywords.split(",").map((item) => item.trim()).filter(Boolean),
        categories: [],
      });
      setTopics((items) => [topic, ...items]);
      setName("");
      setKeywords("");
    } catch (e) {
      setError(e instanceof Error ? e.message : "创建专题失败");
    }
  }

  if (loading) return <p>正在加载专题…</p>;
  return <main><h1>研究专题</h1><form onSubmit={onSubmit}>
    <input value={name} onChange={(e) => setName(e.target.value)} placeholder="专题名称" required />
    <input value={keywords} onChange={(e) => setKeywords(e.target.value)} placeholder="关键词，逗号分隔" required />
    <button type="submit">创建专题</button>
  </form>{error && <p role="alert">{error}</p>}
  {topics.length === 0 ? <p>还没有专题。</p> : <ul>{topics.map((topic) => <li key={topic.id}>{topic.name}</li>)}</ul>}</main>;
}
```

## 5. 常见错误与决策检查

| 现象 | 先检查什么 | 正确处理 |
| --- | --- | --- |
| 新字段未出现在数据库 | 是否只改了 ORM，未生成/执行迁移 | 生成迁移，审阅 SQL，再升级数据库。 |
| 删除专题后论文也消失 | 外键级联错误 | 只级联删除 `topic_papers`，不能删除 `papers`。 |
| Router 中出现 `Session` 或 SQL | 分层被绕过 | 将数据访问移回 Repository，Router 只调用 Service。 |
| API 路径带有 `create`、`build`、`ask` | 将动作当作资源 | 对已有资源使用标准方法；长任务建模为 Job，会话问答建模为 Message。 |
| React 页面刷新后丢失数据 | 数据仅保存在前端 state | 通过 API 保存到 PostgreSQL，再重新加载。 |

本阶段结束后，你应能清楚说明：为什么论文与专题是多对多关系、为什么需要迁移、为什么 Router 不直接操作数据库，以及 RESTful API 如何降低前后端协作成本。

## 补充：阅读状态与会话资源

收藏不足以表达用户是否已阅读论文。为完成研究专题的第一条完整路径，`TopicPaper` 还应保存 `reading_status`，可选值为 `unread`、`reading`、`read` 和 `archived`。状态只属于“专题中的这篇论文”，不能写回全局 `Paper`；同一篇论文在不同专题可以有不同阅读状态。

创建 `app/domain/conversations/models.py`，使问答也成为可恢复的资源：

```python
from datetime import datetime
from typing import Literal
from uuid import UUID, uuid4

from pydantic import BaseModel, Field


class Conversation(BaseModel):
    """表示一个专题中的问答会话。"""

    id: UUID = Field(default_factory=uuid4)
    topic_id: UUID
    title: str = Field(min_length=1, max_length=200)
    created_at: datetime | None = None


class Message(BaseModel):
    """表示会话中的一条用户或助手消息。"""

    id: UUID = Field(default_factory=uuid4)
    conversation_id: UUID
    role: Literal["user", "assistant"]
    content: str
    source_refs: list[dict[str, object]] = Field(default_factory=list)
```

`Literal[...]` 限制 `role` 只能是列出的字符串；`source_refs` 保存回答所用的论文、Chunk、页码和索引版本，不能只保存最终回答文本。

实现顺序是：为 `reading_status`、`conversations` 和 `messages` 创建 ORM 与迁移；用 Fake Repository 覆盖状态更新、专题隔离和消息来源保存；再注册 `PATCH /api/v1/topics/{topic_id}/papers/{paper_id}`、`POST/GET /api/v1/topics/{topic_id}/conversations`、`POST /api/v1/conversations/{conversation_id}/messages`。前端论文列表按阅读状态筛选，并在专题内恢复会话历史。只有这些 API、迁移和页面均通过测试后，才将“标记阅读状态”和“继续追问”列为阶段 1 成果。
