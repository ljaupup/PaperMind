# 工作区协作约定

## Git 规则

- 本仓库仅作本地 Git 管理。不得配置远程仓库、推送或执行其他对外同步操作，除非用户明确要求。
- 所有以 `.` 开头的目录均为本地状态，不得暂存或提交；提交前确认 `.gitignore` 已生效。
- 保留可复现任务所需的源码、配置和关键成果；不要提交虚拟环境、缓存、依赖目录或可重新生成的构建产物。
- 一次提交只覆盖一个任务目录，或一项明确的工作区维护事项；混合变更必须先拆分。
- 提交前依次检查 `git diff --cached --name-only`、`git diff --cached --stat` 与 `git diff --cached --name-status`。
- 提交信息必须使用 `git-commit-message-cn` 技能：采用 `<type>: <简短中文摘要>` 标题，并附 3–5 条基于暂存内容的中文说明。

## 工作方式

- 当前工作空间默认不使用 Superpowers 相关流程或 Skills，也不创建其过程文档；仅在用户明确要求时启用。
- 修改完成后报告实际改动、验证结果与尚未提交或被忽略的内容；不要把计划描述为已完成。

## 阶段学习文档

- `docs/阶段*.md` 先确认用户已明确“学完”的章节；已完成章节及其既有小节是学习记录，不回写正文、代码或命令。对其新问题，只在该章节末尾追加“补充”“勘误”或“增量实现”小节后再进入下一章。只有用户明确允许回溯时才修改，并先说明受影响章节、代码文件和原因。
- 不在阶段文档中写“仅新增”“本节性质”等维护说明。一般答疑不更新文档；只有用户要求补充学习材料时才更新。
- 面向零基础，先解释概念并给出最小可运行示例，再接入 PaperMind。未知第三方 API、参数或返回结构，要在代码块后说明常用字段和用途。
- 代码示例包含必要的单行 Sphinx 文档字符串，不写 `:rtype:`；字段说明放在代码块后，不在代码内堆叠解释字段访问的注释。
- 命令应匹配当日已实现能力，先说明前置条件，避免引用未注册路由、未建表或未启动容器。Docker 和外部服务依次执行检查、拉取或创建、验证。
- 阶段文档修改后检查 Markdown 代码围栏和 `git diff --check`；不自行暂存或提交已有变更。每日提交保持原子化，并按 `git-commit-message-cn` 规则编写提交信息。
